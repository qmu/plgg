# Guide T6 — plgg-view: TEA, typed Html, SSR

## Overview

Document `plgg-view` — the minimal Elm Architecture (TEA) on plgg: the typed
`Html<Msg>` view tree, the element builders, the `sandbox`/`application`
runtimes, and pure `renderToString`/`collectCss` SSR. Use the `example` To-Do app
as the running tutorial.

## Key Files

- `packages/guide/packages/plgg-view/*` (new).
- `packages/plgg-view/` - `Html` model (`el`/`text`/builders, `Attribute`/
  handlers/`transition`), the typed content model (`Phrasing`/`Flow`/`ListItem`,
  cardinality), `mapHtml`, `renderToString`/`collectCss` (SSR), the client
  `render`/`sandbox`/`application` runtimes, `Url`.
- `packages/plgg-view/README.md` - the canonical explanation (TEA scope,
  what-it-is/isn't table, the typed content model).
- `packages/example/src/{app.ts,main.ts,server.ts}` - the worked TEA program.

## Implementation Steps

1. **TEA overview** — `Model`/`Msg`/pure `update`/pure `view` + a tiny runtime;
   the "minimum recognizable TEA" scope (no `Cmd`/`Sub`, no hydration), and why
   it fits plgg's ethos.
2. **The `Html<Msg, T>` tree** — `Box`-union view nodes, the element/text builders,
   static attrs vs event `Handler`s, `mapHtml` as the functor over `Msg`.
3. **The typed content model** — `T` brands the tag; `Phrasing`/`Flow`/`ListItem`/
   void slots and cardinality (`One`/`NonEmpty`/tuples); `el` as the honest escape
   hatch.
4. **Transitions** — `transition`/`fadeIn`/`slideIn` as pure `Anim` data; WAAPI +
   `prefers-reduced-motion`; the SSR-drops-them / client-plays-them boundary.
5. **SSR** — `renderToString`/`collectCss` (pure, no DOM), how plgg-server's
   `htmlDocument` composes them; the **no-hydration** caveat (mount re-renders
   from `init`).
6. **Runtimes** — `sandbox` vs `application` (routing-aware via plgg-router); the
   diff/patch renderer; tie it together with the `example` app as a tutorial.

### Discovered Insights

- **Insight**: plgg-view's typed content model is **producer-side**: a value's
  own type is `Html<Msg, "li">`, so every container accepting `li` enforces it
  automatically — the inverse of React/TSX where each container re-declares its
  `children` constraint. The bare `Html<Msg, string>` (what `el` and `mapHtml`
  produce) is rejected from strict slots by default. Framed in the page as
  guardrails/reach, not ceremony, per the design lens.
- **Insight**: The no-hydration limit is shared across plgg-view, plgg-server
  SSR, and the example — all three say "first paint, then re-render from init,"
  not hydration. Documented identically on all three pages so readers don't
  infer hydration from the SSR/SSG features.
- **Insight**: The example app reflects its UI filter to the URL via
  plgg-router's query codecs (`QueryCodec`/`serializeQuery`/`queryEnum`/…) and
  uses `plgg-view/style` (`sx`) namespaced so Tailwind-style names don't collide
  with element builders — a real pattern worth surfacing, though the guide keeps
  the tutorial page high-level and points at the source.
