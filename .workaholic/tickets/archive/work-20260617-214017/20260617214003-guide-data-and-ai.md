### Discovered Insights

- **Insight**: The plgg-foundry README is **stale** â€” it documents
  `makeFoundrySpec`/`makeProcessorSpec`/`makePackerSpec` and `result.isOk()`
  method-style, but the shipped API is `makeFoundry`/`makeProcessor`/
  `makeSwitcher`/`makePacker` + `runFoundry(foundry)(input)`, with errors folded
  by `matchResult` (errors-as-values). I followed the source and the
  `Example/ProfileFoundry.ts` spec, not the README. The foundry README is worth
  a separate refresh ticket (same situation as the plgg core README's match).
- **Insight**: `makeFoundry` defaults its provider to `openai("gpt-5.1")`, and a
  provider is pure config data (`Box<"OpenAI", Config>` from plgg-kit), not a
  live client â€” so the vendor boundary is a value swap. Documented as the
  selling point per the operation lens.
- **Insight**: `SqlError` (`{ message; cause: Option<unknown> }`) and the core
  `Defect`/`InvalidError` all share the `Box`-union-with-serializable-cause
  idiom; the ticket flags a future shared "boundary error" primitive, which the
  page records as a forward pointer.
