# Publish the guide workload on host port 5181 (match the tunnel)

## Overview

The guide workload publishes host port **5173**, but this server's cloudflared
tunnel routes `plgg-guide.qmu.dev → http://localhost:5181`, so the guide is
unreachable through the tunnel (nothing listens on 5181). The VitePress config
**already** hardcodes `allowedHosts: ["plgg-guide.qmu.dev"]`, and `5173` on this
server also collides with `csnet-poc.qmu.dev` (also mapped to `localhost:5173`
in the tunnel). The fix is to publish the guide on host port **5181** everywhere
it appears, leaving the container-internal port at 5173.

Decision (chosen by the user): **hardcode 5181** rather than make the port
configurable — consistent with the deployment hostname already baked into the
VitePress `allowedHosts`.

## Key Files

- `workloads/guide/compose.yaml` — the published port (`ports: "5173:5173"`,
  line 15) and the header comment URL (line 6). Change the host side to 5181.
- `scripts/serve-guide.sh` — the startup banner URL (line 10). Change to 5181.
- `workloads/guide/README.md` — the "Run it" localhost URL (line 20), the
  plain-docker `-p 5173:5173` (line 36, host side only), and the "Coexists with
  the example demo" port note (line 53). Change host references to 5181.
- `workloads/guide/Dockerfile` — `EXPOSE 5173` (line 24) and the `vitepress dev
  --port 5173` CMD (line 31) are **container-internal**; leave unchanged.
- `packages/guide/.vitepress/config.ts` — already carries
  `allowedHosts: ["plgg-guide.qmu.dev"]`; no change, but it is the precedent for
  baking the qmu.dev deployment detail into the repo.
- `~/.cloudflared/config.yml` — external source of truth (`plgg-guide.qmu.dev →
  localhost:5181`); **do not edit** — the repo aligns to it, not vice-versa.

## Considerations

- **Operation policy — runnable delivery path** (`workaholic:operation` →
  `policies/ci-cd.md`). The launcher must actually reach the documented URL;
  shipping a wrapper that binds a port the tunnel can't see is a broken delivery
  path. This change makes `serve-guide.sh` reach `plgg-guide.qmu.dev` without
  manual per-run fixups.
- **Server-specific value baked into the repo** (`workloads/guide/compose.yaml`).
  Hardcoding 5181 ties the workload to this server's tunnel allocation. The user
  accepted this, and it is consistent with the deployment hostname already
  hardcoded in `packages/guide/.vitepress/config.ts` (`allowedHosts`). If the
  repo is later generalized, the conformant alternative is a configurable port
  (e.g. `"${GUIDE_PORT:-5173}:5173"`) — out of scope here by decision.
- **Container-internal port unchanged** (`workloads/guide/Dockerfile`). Vite/
  VitePress still listens on 5173 inside the container; only the host-published
  port moves to 5181. Do not touch `EXPOSE`/`--port`.
- **Do not edit the tunnel** (`~/.cloudflared/config.yml`). It is the external
  source of truth; pointing it at 5173 would collide with `csnet-poc.qmu.dev`.
- **Existing runtime workaround.** A temporary `docker run` container
  (`plgg-guide-5181`) is currently serving 5181 by hand; once this lands,
  `serve-guide.sh` supersedes it — stop the manual container to avoid a port
  clash (`docker rm -f plgg-guide-5181`).
