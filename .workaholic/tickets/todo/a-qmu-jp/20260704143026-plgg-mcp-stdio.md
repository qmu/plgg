# `plgg-mcp`: a hand-rolled JSON-RPC 2.0 / MCP server on the node stdlib, with a stdio transport and plggpress read-only content tools (`search_content`, `get_article`, `list_collections`)

## Overview

Phase 10 (MCP & plugin), ticket **26** of the plggpress/plggmatic roadmap —
the first of the MCP series. It builds the protocol substrate every later
integration stands on: the HTTP+OAuth transport (ticket 27) and the
auto-generated Claude Code plugin export (ticket 30, which points a
`.mcp.json` at this server). Decision record:
`.workaholic/specs/20260704-plggpress-plggmatic-roadmap.md`.

The governing decision is **D15**, transcribed here so an implementer need not
open the spec: *"MCP implementation — **Hand-rolled** (JSON-RPC 2.0 on node
stdlib) per vendor-neutrality. stdio transport + read-only tools first;
streamable HTTP + OAuth (via our own OP, D6) second."* This ticket is the
**stdio-transport + read-only-tools first** half; ticket 27 is the streamable
HTTP + OAuth second half. No `@modelcontextprotocol/sdk`, no third-party
JSON-RPC library — the whole protocol is written from scratch on `plgg` and
Node stdlib (`process.stdin`/`stdout`/`stderr`), exactly as plgg-md,
plgg-server, plgg-sql, and plgg-auth were written from scratch on the same
foundation. This is the vendor-neutrality / zero-new-deps rule applied to a
protocol: MCP is a wire format, and a wire format is expressible as typed
parse/encode pipelines.

The vision (author's words, condensed in the spec) names *"a native MCP
server"* among plggpress's CMS features, and **D17** ties it to the plugin
export: *"plggpress exports a Claude Code plugin … `.mcp.json` pointing at the
instance's MCP endpoint."* So the stdio server this ticket ships is the exact
process Claude Code will spawn: a team's stored plggpress knowledge becomes
referenceable from Claude Code over three read-only tools —
**`search_content`** (BM25 search over the content index), **`get_article`**
(fetch one document by collection + path), and **`list_collections`** (the
registered content models). These are thin MCP adapters over ticket 16's
in-process query functions — which ticket 16 built HTTP-free precisely so this
server can consume them *"callable in-process as plain typed functions, not
only over HTTP"* (ticket 16 Overview; D17).

**Package split (mirrors ticket 16's plgg-content ↔ plggpress DeliveryApi
seam).** Two layers, one truth:

- **`plgg-mcp`** — a **new, generic, content-agnostic package**: JSON-RPC 2.0
  framing + the MCP lifecycle (`initialize` / `notifications/initialized`,
  `tools/list`, `tools/call`) + a stdio transport loop + a typed `Tool`
  registry. It depends **only on `plgg`** and knows nothing of plggpress,
  plgg-content, HTTP, or SQLite. Registering a tool means handing it a name, a
  description, a JSON-Schema input shape, and a handler
  `(args) => PromisedResult<ToolResult, ToolError>`. This is the reusable
  substrate ticket 27 re-transports over HTTP and ticket 30 packages.
- **plggpress `McpServer/`** — the **adapter**: binds ticket 16's
  `searchIndex` / `getDocument` / `listCollections` (from `plgg-content`,
  already a plggpress dependency after ticket 16) into three `plgg-mcp` tools,
  opens the derived index at startup (`openIndex` + `ingestFromConfig`, D4),
  and runs `plgg-mcp`'s stdio serve loop. Exposed as a **`plggpress mcp`
  subcommand** on the existing CLI (`packages/plggpress/src/cli.ts`) — the
  command a `.mcp.json` `stdio` server entry will invoke.

Why the generic core is its own package and not folded into plggpress: the MCP
server (and the exported plugin) must **not drag in** plggpress's CLI, theme,
or `plgg-bundle` toolchain — the same argument ticket 16 made for keeping
`plgg-content` standalone. Keeping the protocol in `plgg-mcp` also keeps a
**dependency cycle** from forming: plggpress depends on `plgg-mcp`, so
`plgg-mcp` must not depend back — it stays plggpress- and content-agnostic.
Ticket 27 will add an HTTP transport to `plgg-mcp` without touching the
lifecycle/tool core; the split is what makes that a transport addition rather
than a rewrite.

**Resources are deferred to a later pass.** MCP's `resources/list` /
`resources/read` capability is intentionally out of scope here (state it in
the `serverInfo` capabilities so a client sees only `tools`); tools cover the
read-only content surface D15 asks for first. Prompts, tool result streaming,
and pagination cursors on `tools/list` are likewise deferred (Considerations).

The Phase 10 gate this ticket must satisfy (from the spec): *"MCP conformance
exercised with a real client (Claude Code) against stdio …"* — the server is
driven end-to-end by Claude Code over stdio, not merely unit-tested.

## Policies

- `workaholic:operation` / `policies/delivery.md` — this ticket adds a **new
  publishable package** (`plgg-mcp`) and a **new served process** (the stdio
  server). The policy documents that `scripts/build.sh` runs per-package
  `npm run build` in dependency order and is *publish-order authority* (the
  publish order is sed-derived from its `cd`-lines), and that
  `scripts/npm-install.sh` installs each package in order — so introducing
  `plgg-mcp` requires an exact `cd $REPO_ROOT/packages/plgg-mcp && npm run
  build` line and `cd $REPO_ROOT/packages/plgg-mcp && npm install` line placed
  **after plgg core and before plggpress** (plggpress depends on `plgg-mcp`),
  plus a `test-plgg-mcp.sh` line in `check-all.sh`. Get the `cd`-line format
  exactly right or the publish-order derivation drifts.
- `workaholic:design` / `policies/security.md` — the MCP server parses
  **untrusted wire input** on every line of stdin: an attacker (or a buggy
  client) can send malformed JSON, a wrong `jsonrpc` version, an unknown
  method, a `tools/call` naming a non-existent tool, or `arguments` of the
  wrong shape. Every one of these must be handled as a typed JSON-RPC error
  (`-32700` parse / `-32600` invalid request / `-32601` method not found /
  `-32602` invalid params / `-32603` internal), never a throw that kills the
  loop. Tool `arguments` reach the plgg-content query layer **only through
  plgg casters** (the `q`/`limit`/`collection`/`path` fields validated exactly
  as ticket 16's `asListQuery` validates the delivery API's query params) —
  the free-text search term still crosses ticket 15's `fts5Phrase` sanitizer
  inside `searchIndex`, so hostile search input is crash-safe by construction.
  The tools are **read-only** (no write verb, no filesystem mutation), the
  same public content the SSG already serves.
- `workaholic:implementation` / `policies/observability.md` — the policy
  records that *"Log calls are exclusively to the native `console` API"* and
  there is no structured logging framework. For a **stdio transport this is a
  correctness constraint, not a style note**: `stdout` is the JSON-RPC
  channel, so **no diagnostic output may ever touch `stdout`** — every log,
  warning, and `node:sqlite` `ExperimentalWarning`-style notice must go to
  `stderr` only, or it corrupts the protocol stream a client is parsing. The
  single transport seam (`stdioServer`) is the one place `process.std*` is
  touched, and it writes protocol frames to `stdout` and everything else to
  `stderr`.
- `workaholic:implementation` / `policies/test.md` — the 90% four-metric
  coverage doctrine and the "test against the real thing" practice: the
  lifecycle/dispatch fold is exercised with real serialized JSON-RPC frames
  through an **injectable I/O record** (a fake stdin/stdout/stderr, the
  `CliConsole` pattern from `plgg-cli`), and the plggpress adapter's tools run
  against a real `node:sqlite` `plgg-content` index built through ticket 16's
  pipeline. `plgg-mcp` is a **new package gated ≥90 from day one** (D14; ticket
  02 makes a missing `plgg-test.config.json` fail rather than silently skip).

  embedded newline** — the stdio framing invariant).
- `packages/plgg-mcp/src/Mcp/` — the MCP lifecycle + tool layer:
  `model/Tool.ts` (`Tool = { name: Str, description: Str, inputSchema: Datum,

  `ToolResult` = MCP `{ content: [{ type: "text", text }], isError? }`),
  `model/ToolRegistry.ts` (name → `Tool`, with a typed not-found),
  `model/ServerInfo.ts` / `model/Capabilities.ts` / `model/InitializeResult.ts`

  **closed union** `Initialize | Initialized | ToolsList | ToolsCall |
  Unknown` the dispatcher folds with exhaustive `match`),
  `usecase/handleMessage.ts` (request → `Option<JsonRpcResponse>`; `None` for
  the `notifications/initialized` notification), `usecase/dispatchTool.ts`
  (`tools/call`: registry lookup → arg validation → run handler → map

## Implementation Steps

1. **Scaffold and wire `plgg-mcp`.** Create `packages/plgg-mcp` in the plgg
   house layout (mirror `plgg-db-migration`: `package.json` with
   `"type": "module"` semantics of the family, `engines.node ">=22.6"`, a
   `file:` dep on `plgg` only, `plgg-bundle`+`plgg-test`+`@types/node`
   dev-deps, the standard `build`/`test`/`tsc`/`coverage` scripts;
   `tsconfig.json` with the `plgg-mcp*` self-alias `paths` and
   `include: ["src/**/*", "example.ts"]`; `plgg-test.config.json` with a
   **≥90** four-metric `threshold` and `"exclude": ["/index.ts"]`;
   `bin/hook.mjs` copied from a sibling if the toolchain requires it). Its
   `exports` map must include **`types`+`default` under both `import` and
   `require`** (concern 51). Then wire the runner scripts with **exact**
   `cd`-lines:
   - `scripts/npm-install.sh`: insert
     `cd $REPO_ROOT/packages/plgg-mcp && npm install` after the `plgg-content`
     line ticket 16 adds, before the plggpress line (21).
   - `scripts/build.sh`: insert
     `cd $REPO_ROOT/packages/plgg-mcp && npm run build` after ticket 16's
     `plgg-content` build line and before the plggpress line (47). Confirm no
     other package's ordering breaks (nothing but plggpress depends on it).
   - `scripts/check-all.sh`: add `./scripts/test-plgg-mcp.sh` (before the
     plggpress test line) and create `scripts/test-plgg-mcp.sh` +
     `scripts/tsc-plgg-mcp.sh` mirroring `scripts/test-plgg-sql.sh` /
     `scripts/tsc-plgg-sql.sh` verbatim (only the package name changes).
   - Add `plgg-mcp` to plggpress's `dependencies` in
     `packages/plggpress/package.json` (`"plgg-mcp": "file:../plgg-mcp"`) and
     its `bundle.config.ts` externals / `tsconfig.json` paths if the plggpress
     bundler needs the external marked.
2. **JSON-RPC 2.0 model** (`src/Rpc/model/JsonRpc.ts`): `JsonRpcId` =
   `Str | Num`; `JsonRpcRequest` = `{ jsonrpc: "2.0", id: Option<JsonRpcId>,
   method: Str, params: Option<Datum> }` (a notification is a request with no
   `id`); `JsonRpcError` = `{ code: RpcErrorCode, message: Str, data?: Datum }`;
   `RpcErrorCode` a **closed union** of `-32700 | -32600 | -32601 | -32602 |
   -32603` with named constructors. `asJsonRpcRequest(value: Datum)` folds an
   untrusted parsed object into a `Result<JsonRpcRequest, JsonRpcError>`
   (Err `-32600` on a missing/`≠"2.0"` `jsonrpc` or a non-string `method`) —
   plgg casters, no `as`.
3. **Parse / encode** (`src/Rpc/usecase/parseLine.ts`,
   `encodeResponse.ts`): `parseLine` runs `JSON.parse` inside a `Result`
   (a throw → Err `-32700`), then `asJsonRpcRequest`. `encodeResponse`
   serializes a `JsonRpcResponse` to a **single line** with no embedded newline
   (assert this — a `\n` inside a frame breaks stdio framing). Both are pure.
4. **Tool + registry model** (`src/Mcp/model/Tool.ts`, `ToolRegistry.ts`):
   `Tool` = `{ name: Str, description: Str, inputSchema: Datum (a JSON Schema
   object), handle: (args: Datum) => PromisedResult<ToolResult, ToolError> }`;
   `ToolResult` = the MCP `{ content: ReadonlyArray<{ type: "text", text: Str
   }>, isError: Option<Bool> }` shape; `ToolRegistry` wraps a name→`Tool` map
   with a `lookup` returning `Option<Tool>`. `ServerInfo` = `{ name, version
   }`; `Capabilities` advertises `{ tools: {} }` and **omits `resources`**.
5. **Lifecycle dispatch** (`src/Mcp/model/McpMethod.ts`,
   `usecase/handleMessage.ts`): parse the request `method` into the closed
   `McpMethod` union (`Initialize | Initialized | ToolsList | ToolsCall |
   Unknown`) and fold with **exhaustive `match`** —
   - `initialize` → `InitializeResult` (`protocolVersion` echoed/pinned,
     `capabilities` = tools-only, `serverInfo`).
   - `notifications/initialized` → **`None`** (a notification: no response
     frame).
   - `tools/list` → `{ tools: registry.map(t => { name, description,
     inputSchema }) }`.
   - `tools/call` → delegate to `dispatchTool`.
   - `Unknown` → JSON-RPC error `-32601` (method not found).
   `handleMessage` returns `Option<JsonRpcResponse>` so notifications produce
   no frame. Any error rides the `Result`/`Option` channels — never a throw
   escapes to the transport loop.
6. **Tool dispatch** (`src/Mcp/usecase/dispatchTool.ts`): read
   `params.name` + `params.arguments` (validated: missing/garbage → `-32602`),
   `registry.lookup(name)` (`None` → `-32602`/tool-not-found), run
   `tool.handle(args)`, and fold the `Result` into a `ToolResult` — the **Ok**
   branch to `{ content:[{type:"text",text}] }`, the **Err** branch to a
   `ToolResult` with `isError: true` and the error message as text (MCP models
   tool failures as results, not JSON-RPC errors). A protocol-level failure
   (bad params) is a JSON-RPC error; a tool-execution failure is an
   `isError` result — keep the two distinct.
7. **Stdio transport** (`src/Transport/node/stdioServer.ts`): define
   `StdioSeam = { readLines: (onLine: (line: Str) => Promise<void>) => void,
   write: (frame: Str) => void, log: (msg: SoftStr) => void }` and
   `nodeStdioSeam` — the **one** place `process.stdin`/`stdout`/`stderr` are
   touched: `readLines` buffers `stdin` and splits on `\n`; `write` appends
   `\n` and writes to **stdout**; `log` writes to **stderr** (the
   observability-policy correctness rule — nothing but protocol frames on
   stdout). `serveStdio(registry, serverInfo, seam)` wires the loop:
   each line → `parseLine` → on Err, `encodeResponse` an error frame (unless
   the id is absent) → on Ok, `handleMessage` → `matchOption` to write the
   frame or stay silent (notification). Malformed lines never crash the loop.
8. **plggpress content tools** (`packages/plggpress/src/McpServer/usecase/
   contentTools.ts`): given a `plgg-content` `Db`, build three `plgg-mcp`
   `Tool` values:
   - `search_content` — `inputSchema` `{ q: string, collection?: string,
     limit?: number }`; handler validates via ticket 16's `asListQuery`
     semantics and calls `searchIndex`, returning bm25 hits as JSON text.
   - `get_article` — `inputSchema` `{ collection: string, path: string }`;
     handler calls `getDocument`, mapping a typed not-found to an `isError`
     result (not a JSON-RPC error).
   - `list_collections` — no arguments; handler calls `listCollections`.
   Each handler folds the query `Result`/`PromisedResult` into a `ToolResult`;
   no thrown exception escapes; the free-text `q` reaches FTS5 only through
   `searchIndex`'s `fts5Phrase` guard (crash-safe by construction).
9. **plggpress server + CLI command**
   (`packages/plggpress/src/McpServer/usecase/mcpServer.ts` +
   `src/cli.ts`): `mcpServer` opens the derived index at startup via ticket
   16's `openIndex` + `ingestFromConfig` (D4: rebuildable, derived at boot;
   losing the DB costs a rebuild), builds a `ToolRegistry` from
   `contentTools`, and calls `plgg-mcp` `serveStdio` with `nodeStdioSeam`. Add
   an `mcp` command arm to `cli.ts` dispatch (beside build/serve) that invokes
   it; wire it through `bin/plggpress.mjs` (mirroring the
   `plgg-db-migration.mjs` launcher — load the built `dist`, own any dynamic
   config import). Log a one-line "listening on stdio" banner to **stderr**.
10. **Runnable demo** (`packages/plgg-mcp/example.ts`): register a trivial
    tool, drive a scripted `initialize` → `tools/list` → `tools/call` →
    `notifications/initialized` frame sequence through `serveStdio` with a fake
    `StdioSeam` capturing written frames, and print them. The proof-of-value
    artifact (working-style: prove value with a runnable demo). Quote its
    output in the PR.
11. **Specs** (co-located, flat `test()`, absolute imports): in `plgg-mcp` —
    `parseLine.spec.ts` (valid request; bad JSON → `-32700`; wrong `jsonrpc`
    → `-32600`; notification has no `id`), `handleMessage.spec.ts` (initialize
    result shape + tools-only capabilities; `tools/list`; unknown method →
    `-32601`; `notifications/initialized` → no frame — exhaustive `McpMethod`
    coverage), `dispatchTool.spec.ts` (happy call → text content; bad
    `arguments` → `-32602`; unknown tool; a handler returning Err → `isError`
    result, **not** a JSON-RPC error), `stdioServer.spec.ts` (a full

## Considerations

- **Resources deferred by design.** `resources/list` / `resources/read` and
  prompts are intentionally out of this first pass (D15: "read-only tools
  first"); `initialize` advertises only `tools` so a client never offers them.
  Revisit trigger: a consumer that wants to browse the corpus as MCP resources
  (rather than search it) — model `Capabilities` so a `resources` field can be
  added without reshaping the lifecycle.
- **`tools/list` pagination + result streaming deferred.** MCP allows a
  `cursor` on `tools/list` and streamed tool output; with three tools and
  bounded search results neither is needed now. Keep the list response shape so
  a `nextCursor` can be added later (Considerations, not scope).
- **Transport-agnostic on purpose (ticket 27).** The lifecycle and `Tool`
  registry must not know they are on stdio — ticket 27 re-transports the exact
  same `handleMessage`/registry over streamable HTTP + OAuth (via the D6 OP).
  Resist folding any HTTP or stdio assumption into `Mcp/`; keep it in
  `Transport/`. If ticket 27 finds a stdio-ism leaked into the lifecycle core,
  that is a boundary bug to fix here.
- **Index freshness on the served MCP process.** Like ticket 16's delivery
  API, this server builds the index at startup and does not watch the
  filesystem; a long-running MCP process serving a mutated corpus is a
  hot-reload/operations concern (ticket 28, and concern 51's "hot reload does
  not refresh site.config.ts"). For the stdio process (spawned per Claude Code
  session) a boot-time build is correct; the HTTP server (ticket 27) may want
  `syncDocument` on a signal — defer to there.
- **Authorization is out of scope for stdio.** The stdio transport trusts its
  local spawner (Claude Code runs as the user), and the tools expose only
  public read-only content the SSG already serves — so no authz here. The
  moment the server is reachable over the network (ticket 27's HTTP transport),
  OAuth via the D6 OP is mandatory; do **not** half-implement auth in this
  ticket.
- **`node:sqlite` ExperimentalWarning.** The plggpress adapter opens a
  `node:sqlite` index; its `ExperimentalWarning` (accepted status quo in
  plgg-sql/plgg-db-migration/plgg-auth) is written to **stderr** by Node, so it
  does not corrupt the stdout protocol stream — verify this in the transport
  spec's stdout-only assertion rather than suppressing the warning.
- **MCP protocol-version pinning.** Pin the `protocolVersion` this server
  implements and echo/negotiate against the client's `initialize` value; record
  the exact version string in the PR. A future MCP spec revision is a version
  bump here, not a redesign — the lifecycle union already scopes it.
