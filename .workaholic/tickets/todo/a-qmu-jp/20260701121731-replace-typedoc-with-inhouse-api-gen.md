# Replace typedoc + typedoc-plugin-markdown with an in-house API-doc generator

## Overview

The guide's API reference is generated by `packages/guide/scripts/gen-api.mjs`,
which shells out to **`typedoc` + `typedoc-plugin-markdown`** (per package,
against `src/index.ts`) to produce per-symbol markdown, then post-processes it
into compact category pages. These two third-party deps are the last
non-workspace tools in the guide, and ŌĆö per the maintainer ŌĆö a shortcut that
should never have been taken: the monorepo is built on plgg and already depends
on `typescript`, so extracting a package's public API is something we can do
ourselves.

typedoc's ONLY real contribution is, per public symbol: the formatted
**signature** and the **doc summary**. `gen-api.mjs` already derives *which*
symbols are public and *which category* they belong to by statically analysing
each package's source (`export { ŌĆ” }` clauses, `export * from`, `src/<Category>/`
dirs). So the replacement only needs to supply signature + summary per symbol;
the categorisation/compaction/sidebar logic stays.

Recommended approach ŌĆö **parse the `.d.ts` plgg-bundle already emits**: the
per-file declarations under each package's built `dist` ARE the resolved,
tsc-formatted public surface (`export type Cause = Readonly<{ŌĆ”}>;`,
`export declare const cast: (ŌĆ”) => ŌĆ”;`), with JSDoc preserved. The build order
already builds every dist before `docs:api` (dev Dockerfile / deploy CI), so the
`.d.ts` are on disk. Read them with the TypeScript compiler API (already a
dep) ŌĆö `ts.createSourceFile` per `.d.ts`, walk the export statements, take each
declaration's printed text as the signature code block and its leading JSDoc as
the summary. This reuses existing in-house output and adds **zero** new deps.

## Scope

- Remove `typedoc` and `typedoc-plugin-markdown` from `packages/guide/package.json`.
- Replace the typedoc invocation in `gen-api.mjs` with an in-house extractor.
- Keep the generated `api/<pkg>/*.md` output shape and `api/typedoc-sidebar.json`
  manifest equivalent (same categories, compact per-symbol entries), so
  `plgg-press` renders the guide unchanged.
- Add a no-typedoc gate (mirrors `scripts/gate-vite.sh` / `scripts/gate-happy-dom.sh`).

## Implementation Steps

1. Build the in-house extractor over the emitted `.d.ts` (TS compiler API):
   collect public exports of a package, honouring `@internal`/private exclusion
   and the `Abstracts`/`Grammaticals` dir exclusion; produce `{name, kind,
   signature, summary}`. The `signature` is the declaration's printed `.d.ts`
   text (strip `declare`/`export` noise to match the current compact form); the
   `summary` is the leading JSDoc's first line/paragraph.
2. Rewire `gen-api.mjs` to feed those records into its existing categorisation +
   compaction, replacing the typedoc call. Emit the same `api/<pkg>/*.md` and
   `api/typedoc-sidebar.json` structure (rename the sidebar file only if a clean
   name is warranted and the site config is updated in lockstep).
3. Delete `typedoc.base.json`; remove the two deps from `guide/package.json`;
   regenerate the guide lockfile.
4. Regenerate the API docs (`npm run docs:api` in guide) and DIFF against the
   committed `api/**` ŌĆö resolve any parity gaps (missing summaries, signature
   formatting).
5. Add `scripts/gate-typedoc.sh` and wire it into `check-all.sh` + CI.
6. Verify: guide builds (`plgg-press build`) and renders the API pages; the
   `verify-render.mjs` golden check (if present) passes; `grep -rn "typedoc"
   packages` is clean.

## Considerations

- **JSDoc preservation in `.d.ts` is the make-or-break risk.** `tsc` keeps JSDoc
  in declarations by default, but confirm plgg-bundle's `emitDts` does NOT set
  `removeComments`. If comments are stripped, either flip that flag or fall back
  to running the extractor over `src/index.ts` (compiler API + type checker:
  `symbol.getDocumentationComment`, `checker.typeToString`) instead of `.d.ts`.
  Decide the source (dist `.d.ts` vs. source) at the /drive gate.
- **Parity, not pixel-identity.** typedoc reconstructs signatures its own way;
  the `.d.ts` is tsc's formatting. The bar is: every public symbol still listed,
  correct category, readable signature + one-line summary. Diff the regenerated
  `api/**` against the committed output and eyeball a few pages; small formatting
  deltas are acceptable, missing symbols/summaries are not.
- **Re-exports.** `gen-api.mjs` already resolves `export * from "<other-pkg>"`
  and `export { ŌĆ” } from "<pkg>"` for categorisation; the extractor must surface
  re-exported symbols' signatures too (they live in the re-exporting package's
  `.d.ts` or the target's ŌĆö follow the same resolution gen-api uses).
- **Zero new deps** (vendor-neutrality): reuse the already-present `typescript`;
  do NOT add a markdown/AST lib. House style (no `as`/`any`/`ts-ignore`) applies
  to any `.ts` added; note `gen-api.mjs` is a plain `.mjs` script (not `packages/`
  TS) so it follows script conventions, but a new `.ts` module does.
- **Serving unaffected.** The `api/**` markdown is committed and rendered by
  plgg-press; typedoc only runs at `docs:api`/build time, so this never touches
  the served site's runtime ŌĆö only regeneration.
- Add the no-typedoc gate as one canonical script shared by `check-all.sh` and
  CI (the vite/happy-dom-gate pattern), not a bespoke per-package script.
