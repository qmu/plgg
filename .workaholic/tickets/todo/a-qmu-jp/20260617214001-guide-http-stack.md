# Guide T5 — HTTP stack: plgg-http, plgg-router, plgg-server, plgg-fetch

## Overview

Document the HTTP family — the runtime-neutral model, the routing toolkit, the
server, and the symmetric client — as one cohesive section, since they share a
vocabulary (`HttpRequest`/`HttpResponse`/`HttpError`) and compose together.

## Key Files

- `packages/guide/packages/{plgg-http,plgg-router,plgg-server,plgg-fetch}/*` (new).
- `packages/plgg-http/` - `HttpRequest`/`HttpResponse`/`HttpStatus`/`Method`/
  `HttpError`/`ResponseBody`, the shared base.
- `packages/plgg-router/` - `compile`/`match` patterns, `Segment`, query parse.
- `packages/plgg-server/` - `web`/`get`/`post`/`route`, middleware onion,
  `Context`, `handle`/`toFetch`, View (`htmlDocument`/`pageResponse`/SSR), **SSG
  (`generateStatic`, `plgg-server/ssg`)**, runtime adapters (node/bun/deno).
- `packages/plgg-fetch/` - the typed client, symmetric with the server.
- All four `README.md`s - primary source.

## Implementation Steps

1. **plgg-http** — the shared model page: the plgg-native request/response types,
   `HttpError` as a `Box` union, `ResponseBody` (`SoftStr | Bytes | Stream`).
2. **plgg-router** — compiling and matching path patterns
   (`Static`/`Param`/`Wildcard`), query parsing; how plgg-view's `application`
   runtime consumes it.
3. **plgg-server** — the largest page set: building a `Web` as a data-last `pipe`,
   verb helpers + groups, the middleware onion (`use`, `Context`, state), `handle`
   vs `toFetch`, the View layer (`htmlDocument`/`pageResponse`, SSR), the **SSG
   generator** (`generateStatic`, crawl-via-`handle`, directory-index output,
   the `node:fs` seam), and the runtime adapters.
4. **plgg-fetch** — the typed client: request building, response decoding with
   `cast`, the `matchResult` consumption idiom (no `proc` on the client per the
   style guide), symmetry with the server model.
5. **End-to-end example** — point to `packages/example` (SSR + CSR To-Do) as the
   worked example that ties routing + View + SSR together; reference the SSG
   demo (`example/src/build.ts`).

## Considerations

- **SSG and the error model are current.** Document `generateStatic`'s
  `Result<…, SsgError | Defect>` shape and the `plgg-server/ssg` node-only entry
  exactly as shipped. (`packages/plgg-server/src/Ssg/`)
- **Runtime neutrality is a selling point.** Emphasize that the core imports
  nothing runtime-specific and `toFetch` runs on Workers/Deno/browser — an
  `standards:operation` (vendor-neutrality) highlight.
- Largest content ticket; if it proves too big at drive time, the server page
  set can split from http/router/fetch — keep page boundaries clean so a later
  split is mechanical.
- Pull request/response/route examples from the packages' specs and `example.ts`.
