# Add plgg-router — a lightweight SPA client-side router on top of plgg-view

## Overview

Today plgg has a complete SSR + JSON + CSR loop (plgg-view + plgg-server + plgg-fetch + plgg-sql, demoed in `src/example/`), but the CSR side has **no client-side route changing** — `src/example/src/client.tsx` hydrates a single component tree at `#root` and re-renders on data mutations via event delegation. To build a real SPA on plgg-view, the missing piece is a router: a function from `Location → VNode`, plus the History-API integration that turns `<a>` clicks and `popstate` events into re-renders.

This ticket adds **plgg-router**, a new experimental package under `src/plgg-router/`, exposing the smallest useful surface:

- A data-last `Router` builder mirroring plgg-server's `Web` builder — `router()`, `get(path, handler)`, `route(basePath, sub)` — but where `handler: (loc: Location) => VNode` (no HTTP methods, no `Result`/`HttpError`).
- Path matching with `:param` and `*wildcard` segments, semantically identical to plgg-server's matcher.
- `param`/`query` accessors returning `Option<SoftStr>` (plgg vocabulary, never raw strings).
- A `./client` subpath that wires the History API: `start(router, container)` subscribes to `popstate`, intercepts in-app `<a>` clicks (preserving modifier-clicks, `target="_blank"`, `download`, `rel="external"`, and non-`http(s)` schemes), and re-invokes `render(view, container)` on every location change. `push(path)` / `replace(path)` programmatic navigation.
- That's it — no middleware, no guards, no lazy loading, no hash routing in v1.

### Why a separate package

plgg-server's `Routing` module has all the pure path machinery (`Segment`, `compilePattern`, `matchSegments`, `splitPath`, `compileRoutes`), but the **architecture dependency-direction rule** forbids plgg-router from importing it: peer experimental packages cannot import each other's internals, only plgg core and immediate dependencies (here: plgg + plgg-view). The pure path files are ~150 lines total; **cloning is cheaper than coupling** and mirrors the project's existing pattern (plgg-fetch re-declares `HttpError` shape rather than importing from plgg-server). A follow-up ticket can lift `Segment` + matcher into plgg core if drift becomes a problem.

### Naming note

The word "router" has package-name history in this repo: `plgg-web → plgg-http-router → plgg-server`. The bare name `plgg-router` is now free, but the renaming order means consumers may still associate "router" with server routing. The README must open with a one-paragraph clarification: **plgg-server's `Routing` module is server-side path → `HttpResponse` matching; plgg-router is client-side path → `VNode` matching; the two share `Segment`/`:param`/`*` vocabulary by parallel definition so a reader of both packages sees the same words.**

### Out of scope (deferred to follow-up tickets)

- Hash-based routing strategy (History API only in v1).
- Navigation guards / middleware (no `use()` analogue).
- Lazy-loaded routes (no async handlers).
- Nested `<Outlet>` / layout routes.
- Scroll restoration policies beyond browser-default.
- Server-side hydration handoff (matching the SSR-resolved route on first paint without a re-fetch). The example demo hydrates once, then routes client-side — SSR-router parity is a future story.
- Lifting `Segment`/matcher into plgg core.

New file (plgg-server parses query via Web `URL`; for parity in a pure-data context, plgg-router does it directly).
- `src/plgg-router/src/Routing/usecase/resolve.ts` — `resolve(router: Router, loc: Location): Option<VNode>`. Iterates `router.routes` in registration order; first `matchSegments` hit wins; returns `none()` on no match. Folds matched params into `loc.params` before calling the handler. Optionally memoize compiled routes via WeakMap (mirrors plgg-server's `compileRoutes`); skip for v1 — flat scan is fine for typical SPA route counts.
- `src/plgg-router/src/Routing/usecase/param.ts` — data-last `param(name: SoftStr): (loc: Location) => Option<SoftStr>` and `query(name: SoftStr): (loc: Location) => Option<SoftStr>`. **Vocabulary parity with `src/plgg-server/src/Http/model/Context.ts` `param` / `query`.**
- `src/plgg-router/src/Routing/usecase/client.ts` — the DOM/History seam (the only file imported via the `./client` subpath barrel). Exports:
  - `currentLocation(): Location` — reads `window.location` (pathname + search), parses query, returns a Location with empty `params` (resolve fills them).
  - `push(path: SoftStr): void` — `window.history.pushState(null, "", path)` then dispatch a synthetic event the started router subscribes to.
  - `replace(path: SoftStr): void` — same with `replaceState`.
  - `start(router: Router, container: Element, options?: StartOptions): () => void` — wires `popstate`, intercepts clicks, calls `resolve(router, currentLocation())` and `render(view, container)` on every change. Returns a cleanup function that unsubscribes everything. Options: `{ render?: (vnode: VNode, container: Element) => void; notFound?: VNode }` — by default uses `plgg-view`'s render? No — `plgg-view` doesn't expose a DOM render (that's in `plgg-server/client`'s `render`). To keep plgg-router free of plgg-server dependency, **the consumer passes `render` via `StartOptions`** (the example wires `import { render } from "plgg-server/client"; start(router, root, { render })`). This is a deliberate dependency-direction choice — see Considerations.
- `src/plgg-router/example.ts` — runnable demo (mirrors `src/plgg-server/example.ts`). Builds a 3-route router (`/`, `/users/:id`, `*` not-found), wires it to a happy-dom container in a script (`npx tsx src/plgg-router/example.ts` — runnable only as a sketch under happy-dom or via the example/ integration below).

## Considerations

- **Dependency-direction rule forbids plgg-router → plgg-server.** Per `.workaholic/constraints/architecture.md` (Package Dependency Direction) and the policy discovery summary, peer experimental packages cannot import each other's internals. This is why `Segment`/`compilePattern`/`matchSegments` are **cloned** from plgg-server rather than re-exported, and why `StartOptions.render` is **injected by the consumer** rather than imported from `plgg-server/client`. The example wires `import { render } from "plgg-server/client"; start(router, root, { render });` itself. Long-term, the pure path machinery and the `render(VNode, Element)` signature could lift into plgg core — flag as a follow-up if drift becomes a problem. (`src/plgg-router/src/Routing/model/Segment.ts`, `src/plgg-router/src/Routing/usecase/client.ts`)
- **Naming history.** `plgg-router` was indirectly preceded by `plgg-http-router` (now `plgg-server`). The README must open with the disambiguation (`src/plgg-router/README.md`). A note in the root `README.md` that "plgg-server's `Routing` is server-side; plgg-router is client-side; vocabulary parallels" helps future readers.
- **Accessibility — this is the project's first UI package.** `.workaholic/policies/accessibility.md` currently says "not applicable" because plgg has no UI. plgg-router breaks that premise. v1 obligations (encode these in `client.ts` and document in README):
  - Preserve browser default for: modifier-clicks (`Ctrl`/`Cmd`/`Shift`/`Alt`), non-left-button clicks, `target` attributes, `download` attribute, `rel="external"`, `rel="noopener"` or `rel="noreferrer"`, cross-origin hrefs, non-`http(s)` schemes (`javascript:`/`mailto:`/`tel:`/`data:`).
  - Use History API only (no hash routing) — the modern, accessible default.
  - Handle `popstate` for back/forward.
  - Recommended but **deferred to follow-up**: focus management after navigation (move focus to a route landmark for screen-reader users), `aria-live` route-change announcement, `document.title` policy. Note the deferral explicitly in the README so it isn't mistaken for a missed requirement.
  Reference `src/plgg-router/src/Routing/usecase/client.ts` (intercept-click guards) and `src/plgg-router/README.md` (policy section).
- **Security — preserve browser-default link semantics.** Per `standards:leading-security`, the link-interception must not break browser security UX: `target="_blank"`, `download`, `rel="external"`, `javascript:`/`mailto:`/`tel:` must pass through untouched. `push`/`replace` must reject non-`http(s)` URLs (silently no-op or throw — recommend silent no-op so misconfigured callers don't crash an app, but flag in the spec). No `eval`, no URL-as-code interpretation. (`src/plgg-router/src/Routing/usecase/client.ts`)
- **Coverage strategy for happy-dom code.** Existing precedent (`src/plgg-server/src/View/usecase/render.spec.tsx`) shows the `// @vitest-environment happy-dom` directive is sufficient — no special vitest config needed. The `client.ts` file has more imperative DOM interaction than `render.ts`; budget extra spec time. If a specific branch (e.g., a browser-only API happy-dom doesn't support) can't be covered, add it to `vite.config.ts coverage.exclude` and note why in a code comment, not via `/* v8 ignore */`. (`src/plgg-router/src/Routing/usecase/client.spec.ts`)
- **Zero third-party runtime deps.** Per `.workaholic/specs/infrastructure.md` ("only runtime dependency is plgg") and the project memory's plgg working style, plgg-router's `dependencies` must contain only `plgg` and `plgg-view`. No `history` package, no `path-to-regexp`, no `qs` — the spec discovery confirmed the pure path machinery is ~150 lines, and `parseQuery` is another ~20 lines. (`src/plgg-router/package.json`)
- **Build order matters.** `sh/build.sh` builds packages so their `file:` dependents can resolve `dist/`. plgg-router depends on plgg + plgg-view; insert it after both. The example/ integration (if pursued in a follow-up) would also depend on plgg-router, so plgg-router's build slot is before example. (`sh/build.sh`, `sh/check-all.sh`)
- **Coverage exclusions.** `index.ts` and `src/client.ts` are pure barrels; exclude them (matches plgg-server's precedent). Do NOT exclude `Routing/usecase/client.ts` — its logic must be covered. (`src/plgg-router/vite.config.ts`)
- **No `as`/`any`/`ts-ignore`.** Especially watchful around the click-interception code: `event.target` is `EventTarget | null`, narrowing it to `HTMLAnchorElement` via `instanceof HTMLAnchorElement` (and walking up `parentElement` with the same check) avoids any cast. The `Location` model's `Dict<string, SoftStr>` should round-trip through `pipe(fromNullable, getOr(""))` for missing query keys. (`src/plgg-router/src/Routing/usecase/client.ts`)
- **History API surface — minimal v1.** `push` and `replace` are the only two write operations. Don't expose `history.back()` / `history.forward()` / `history.go(n)` — they're already global. Don't model "navigation events" as a value (no `Result<Location, NavigationError>`); v1 navigations always succeed (or are silently ignored as no-ops). If guards or async loads become a need, that's a follow-up that introduces `proc` chains.
- **`Web` builder shape kept symmetric with plgg-server.** The user-facing names — `get(path, handler)`, `route(basePath, sub)`, `param(name)` — must match plgg-server's exactly. Drop only what's HTTP-specific (`post`/`put`/`patch`/`del`/`head`/`options`, `use(middleware)`). This is the whole point of the parallel-definition strategy. (`src/plgg-router/src/Routing/model/Router.ts`)
- **No SSR-router handoff in v1.** A real SPA hydrating from SSR would want the server to render the matched route on first paint, then plgg-router takes over. That requires either (a) plgg-server importing plgg-router's resolve (blocked by dep direction) or (b) the host wiring both manually. Defer entirely; flag as a future story. The v1 example demo is CSR-only.
- **Tests cannot run real browser navigation.** happy-dom's `window.history.pushState` mutates `window.location` correctly but does not navigate. Specs verify state changes and listener invocations, not visible page changes. Note in the test file headers.
- **Cross-origin `<a>` guard.** Inside `interceptClick`, compute `new URL(anchor.href, window.location.href)` and compare `origin` to `window.location.origin`; only intercept if equal. happy-dom supports this. (`src/plgg-router/src/Routing/usecase/client.ts`)
- **Plan-mode artifacts.** The user invoked `/ticket` with "plan" in the description. This ticket is the plan. No code is being written here. `/drive` implements it.

## Open Questions (decide during implementation, document in Final Report)

- Whether to memoize compiled routes via WeakMap (mirrors plgg-server's `compileRoutes`). Recommend skip for v1 — typical SPAs have <50 routes; a flat scan over an array of pre-compiled segment lists is fine.
- Whether to expose a `currentLocation` *signal* (something handlers can subscribe to for re-render) or keep re-render the host's responsibility (it already is — `start` calls `render` after every change). Recommend: keep the host model. A signal/observable is a larger primitive that belongs in plgg core, not in plgg-router v1.
- Whether `push`/`replace` reject non-`http(s)` URLs (throw vs. no-op). Recommend: silent no-op, log nothing. Misuse is a programming error; throwing would crash the app, which is worse than the navigation simply not happening.
- Whether to include `scrollTo(0,0)` on `push`. Recommend yes (browser default for clicked links); document it.
- Whether to ship a minimal `<Link>` component in plgg-view-style JSX as a convenience. Recommend defer — `<a href>` works fine with the click interceptor; a `<Link>` component is sugar that can come later without API breakage.
