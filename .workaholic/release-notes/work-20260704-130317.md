# Complete the plggpress/plggmatic product roadmap

## Summary

Completes a 20-ticket, 11-phase roadmap building plggpress as a browser-based CMS and plggmatic as its declarative UI framework. Delivers a type-driven design system, content delivery with search, authentication, DB-primary content domains, a voice agent, a hand-rolled MCP server, and production operations — vendor-neutral and zero-dependency where possible.

## Key Changes

- Declarative UI framework (plggmatic): type-driven Declare vocabulary, pure scheduler, and a mode-agnostic Scene renderer (multi-column, single-column, admin views).
- Design system: role×variant token matrix, non-color scales, palette-override API, framework-owned scheme persistence, and a full plggpress theme rewrite onto plggmatic.
- Content platform: SQLite FTS5 search, a MicroCMS-like delivery API, and a zero-dependency RAG search layer.
- Authentication and admin: OIDC login, password accounts, role-based admin access, and CSRF-protected SSR forms.

## Changes

### Added
- plggmatic declarative UI framework: vocabulary, scheduler, and Scene renderers.
- plgg-content delivery API with SQLite FTS5 search and zero-dependency RAG.
- OIDC authentication with password accounts, roles, invites, and sessions.
- DB-primary content domains: stakeholder, guest drafts, and media assets.
- Voice agent over OpenAI Realtime and a hand-rolled MCP server with Claude Code plugin export.
- Production operations: health endpoint, hot WAL backup, and operations runbook.

### Changed
- plgg-view reworked into a full TEA runtime (`update` returns commands, runtime gains subscriptions) — breaking change carried through all consumers.
- plggpress theme cut over to plggmatic design tokens.
- plgg-bundle vendor boundary refactored (`node:`/`tsc` confined to `vendors/`).
- Coverage gate flipped to opt-out with a 90% default.

### Removed
- The stale `plgg-press` remnant.
- plggmatic's errant `publish` npm lifecycle script that broke the release flow.
