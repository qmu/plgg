# Make two plgg-auth crypto tests runtime-portable to unblock main

## Summary

PR #54 merged the plgg-auth OIDC provider to `main`, but its fresh-clone `Run Tests` CI went red: two tests forced a signing failure with a degenerate RSA JWK, which the CI runner's OpenSSL build imports lazily and signs successfully instead of rejecting. This change replaces the fragile bad-key-material trigger with a deterministic, spec-mandated wrong-usage failure that WebCrypto raises identically on every runtime, restoring a green `main` once landed.

## Key Changes

- Replace the non-portable degenerate-key signing-failure trigger with a wrong-usage one that fails deterministically across OpenSSL builds
- Sign the `SignFailure` test with a verify-only `CryptoKey` (`InvalidAccessError`)
- Drive issuance's `ServerError` path with a `use:"enc"` signing key the import rejects (`DataError` → `KeyFailure`)
- Extract `signWith(cryptoKey)` from `signJws` as an honest import/sign seam
- Add the optional RFC 7517 `use` field to `RsaPrivateJwk`, emitted only when set

## Changes

### Added
- `signWith(key: CryptoKey)(input)` in the Jose layer — signs a prepared `header.payload` input with an already-imported RS256 key; `signJws` now imports the JWK and delegates to it
- Optional `use?: "sig" | "enc"` (RFC 7517 §4.2) on `RsaPrivateJwk`, serialized by `privateJwkJson` only when present

### Changed
- The `SignFailure` and `ServerError` coverage tests now trigger their crypto failures through wrong key usage (validation errors raised before any key-material math) instead of a structurally-valid-but-degenerate key whose acceptance is lazy and runtime-specific — both defensive branches stay covered on every runtime
