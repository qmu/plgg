# plgg-md accepts the frontmatter we actually write, opens the heading seam, and fixes three grammar bugs

## Summary

`plgg-md`'s YAML subset rejected **71% of this repository's own front matter** — 472 of
661 files, including the ticket that reported it. It now rejects one, and that one is a
genuine block scalar the subset deliberately excludes. Shipped as **`plgg-md@0.0.3`**.

Alongside it: the heading element is now an injectable seam, and three real grammar bugs
are fixed. Nothing that renders today moved — `golden.spec.ts` stayed green across the
whole branch, and both the guide and strategy corpora are byte-identical.

## Key Changes

- **The YAML subset admits two spellings it already modelled.** `layer: [UX]` and a bare
  `commit_hash:` are standard ticket shape and carry no expansion, aliasing, or type
  coercion. Flow sequences and mappings of scalars are now accepted, and an empty value is
  `YNone` — which `foldYaml` **omits**, so `forOptionProp` reads it as `None` with no
  YAML-specific handling. Corpus rejections: **472 → 1**.
- **The bound got tighter, not looser.** `[` and `{` remain excluded for *scalars*, and
  that is now what bounds nesting: every flow element and every block `- ` item routes
  through the same classifier, so `[[a]]`, `[{a: 1}]`, and `- [a]` are refused with no
  depth counter anywhere. Anchors, tags, merge keys, and block scalars stay shut.
- **`decorateHeading` seam.** `renderHeading` was unreachable, so heading numbering could
  not be built on `plgg-md` without reimplementing block rendering. A site can now own the
  whole `<hN>`. `plgg-md` computes the heading's `Ordinal` and hands it over — a decorator
  holds no state, so a numbering scheme cannot drift from the table of contents. The
  default renders byte-identically to before.
- **Three grammar bugs fixed.** A code fence's closer must now be at least as long as its
  opener (CommonMark §4.5), so the standard idiom of a longer fence embedding a shorter
  one works. A container's colon scan is opaque to fenced blocks, so a callout documenting
  `:::` syntax no longer closes on its own code sample. A table body row must lead with a
  pipe, so prose below a table is no longer absorbed into it.

## Upgrade Notes

- **`RenderOptions` gained a required `decorateHeading` field** — breaking for anyone
  constructing one by hand. `renderOptions()` and `renderMarkdown()` are unaffected.
  Published `plggpress@0.0.4` depends on `plgg-md@^0.0.2`, and a caret on a `0.0.x` pins
  the patch, so it cannot resolve to `0.0.3` and cannot break; a plggpress release needs
  its own bump to pick this up.
- **Table rows are a deliberate dialect divergence from GFM.** Measured against
  markdown-it (what VitePress renders with), a paragraph after a table is a row —
  `plgg-md` previously matched that. It no longer does, by choice: in a bounded authored
  subset, prose joining the table above it is never what the author meant. The cost is
  that pipe-less GFM rows (`1 | 2` under `A | B`) are prose here. This is recorded in
  `tableBodyLine`'s doc comment, not left to be rediscovered.
- **The normative spec moved with the code.** `Yaml/model/YamlValue.ts`'s doc comment is
  the subset's contract; read it before widening anything further.
