# Minimal Elm Architecture for plgg-view + plgg-http extraction + HTTP package renames etc

## Summary

A monorepo-wide modernization: plgg-view's passive VNode/JSX runtime is replaced clean-slate with a minimal Elm Architecture (a typed `Html<Msg>` tree, pure `sandbox`/`application` runtimes, and SSR `renderToString`); the HTTP packages are renamed and re-layered so their shared model lives in a new neutral `plgg-http` that both plgg-server and plgg-fetch build on; and the repo is restructured, style-codified, and brought into consistency. `scripts/check-all.sh` is green end-to-end.

## Key Changes

- Replaced plgg-view's VNode/JSX runtime with a minimal Elm Architecture — `Html<Msg>` hyperscript (no JSX), pure `sandbox`/`application` runtimes, SSR `renderToString` — and migrated plgg-server's View to it
- Extracted the shared HTTP model into a new neutral `plgg-http`, so plgg-fetch no longer depends on plgg-server (they become true peers)
- Renamed `plgg-http-router` → `plgg-server` and `plgg-http-client` → `plgg-fetch`, and added non-Node runtime adapters (`./node`/`./bun`/`./deno`)
- Added `plgg-router` (a pure SPA path toolkit) and reshaped the example into an SSR+CSR To-Do app driven by one TEA program
- Restructured the repo (`sh/`→`scripts/`, `src/`→`packages/`), authored a `plgg-coding-style` skill, made plgg-foundry/plgg-kit compliant, and unified `package.json` exports

## Changes

### Added

- `plgg-view` minimal Elm Architecture: `Html<Msg>`/`Attribute<Msg>` model + hyperscript builders, `foldHtml`/`mapHtml`, pure SSR `renderToString`, and a `./client` runtime (`sandbox`, `application`, DOM `render`)
- `plgg-http` package: the runtime-neutral HTTP model (`Method`, `HttpStatus`/`statusOf`, `HttpRequest`, `HttpResponse`/`ResponseBody`, `HttpError` + builders/matchers) shared by plgg-server and plgg-fetch
- `plgg-router`: pure client-side path toolkit (`compilePattern`/`matchSegments`/`parseQuery`/`param`/`query`) consumed by plgg-view's `application` runtime
- `plgg-coding-style` Claude Code skill; non-Node server adapters (`./node`/`./bun`/`./deno`)
- An SSR + CSR To-Do example rendered from a single Elm-Architecture program

### Changed

- Renamed `plgg-http-router` → `plgg-server` and `plgg-http-client` → `plgg-fetch` (history-preserving)
- plgg-server's `View` now renders plgg-view's `Html`; its HTTP model is re-exported from `plgg-http`
- plgg-fetch now depends on `plgg-http` instead of plgg-server
- Restructured repo layout (`sh/`→`scripts/`, `src/`→`packages/`); unified non-core `package.json` exports to the `.`-keyed shape
- Brought plgg-foundry and plgg-kit into coding-style compliance (removed `as`/`throw`/mutable state; added offline tests)

### Removed

- The legacy plgg-view VNode model, `foldVNode`, and the jsx-runtime/jsx-dev-runtime entries
- plgg-router's `Handler`/`Route`/`Router`/`resolve` and its DOM/History `client` seam (subsumed by plgg-view's `application`)
- plgg-server's redundant client DOM renderer and `./client` subpath
