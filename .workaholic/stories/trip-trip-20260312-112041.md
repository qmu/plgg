# Enforce Proper Layering Between PLGG Foundation and Extension Libraries

## Overview

A collaborative refactoring trip to reorganize the PLGG monorepo, enforcing clean dependency direction between the foundation library (PLGG), vendor integrations (PLGG Kit), and AI pipeline orchestration (PLGG Foundry). The trip eliminated all `as` type assertions and `@ts-ignore` directives from source files, decoupled Foundry from Kit, and established a proper three-layer architecture.

## Motivation

PLGG is a pure TypeScript foundation library providing functional programming primitives — Result, Option, Box, cast, proc, pipe, match. Two extension libraries build on it: PLGG Kit (LLM provider abstractions) and PLGG Foundry (AI workflow orchestration). However, Foundry had a lateral dependency on Kit, 11 `as` type assertion violations existed in Foundry source files, and a `@ts-ignore` sat in the foundation's HKT system. The codebase needed structural investment to enforce its intended layering before these issues compounded.

## Journey

### Planning Phase: Three Perspectives Converge

Three agents — Planner (business vision), Architect (structural coherence), and Constructor (technical accountability) — independently analyzed the codebase and produced artifacts:

- **Direction** (Planner): Framed the refactoring as a structural investment for four stakeholder personas — consumers, contributors, extension authors, and the project owner. Defined "well-organized" as traceable, predictable, separable, and minimal.

- **Model** (Architect): Discovered 10 structural gaps (G1-G10) through comprehensive codebase analysis. Mapped the full dependency graph and proposed a target architecture where Foundry and Kit become peer layers, both depending only on the PLGG foundation.

- **Design** (Constructor): Proposed a 4-phase implementation plan targeting all `as` casts, the `@ts-ignore`, and internal type improvements.

### The Decoupling Debate

The mutual review phase surfaced one significant disagreement: should Foundry's dependency on Kit be decoupled now or deferred?

- **Architect** argued for immediate decoupling via a ~30-line `GenerateAlignmentFn` abstraction type
- **Constructor** argued for deferral — the dependency follows correct layering direction and decoupling would break the public API
- **Planner** initially sided with Constructor on deferral

Planner mediated. After examining the Architect's concrete proposal, Planner reversed position: the effort was minimal (~30 lines), the benefit real (clean dependency direction), and v0.0.1 with an "UNSTABLE" banner was the right time for this structural fix. The team also agreed on narrowing Phase 1 from 23 factory functions to just 3, using the `unsafe*` naming prefix to signal throwing behavior.

### Implementation: Six Phases

Constructor implemented all changes across six phases, each independently compilable:

1. **Foundation unsafe constructors**: Added `unsafeStr`, `unsafeKebabCase`, `unsafeBool` to PLGG — throwing factories that validate and return unwrapped types
2. **RegisterEntry typing**: Replaced `Param = unknown` with a proper `RegisterEntry` type, giving `Env` and `operate.ts` structural type safety
3. **Cast elimination**: Replaced all 8 `as` type assertions in Foundry with `unsafe*` function calls
4. **Foundry-Kit decoupling**: Defined `GenerateAlignmentFn` in Foundry, removed Kit dependency, implemented `asProvider` and `asGenerateAlignmentFn` adapter in Kit
5. **Deduplication**: Extracted `formatEntries` from Processor, Switcher, and Packer into shared Apparatus module; deleted stray `TodoFoundry.ts`
6. **@ts-ignore removal**: Replaced with `@ts-expect-error` for MapKind3's unused type parameters

### Verification

Architect performed analytical review (APPROVE WITH NOTES — minor import ordering and stale JSDoc). All 8 E2E validation scenarios passed:

- Full system coherence (`check-all.sh`): 338 plgg tests + 6 foundry tests green
- Zero `as` assertions in source files
- Zero `@ts-ignore` in source files
- Foundry has no dependency on Kit
- All new exports (`unsafe*`, `asGenerateAlignmentFn`, `asProvider`) properly available
- No stray files or duplicated helpers

## Highlights

- **Consensus through structured disagreement**: The decoupling debate produced a better outcome than either initial position. The Architect's concrete proposal changed the Planner's cost/benefit analysis.
- **Narrow scope, full coverage**: Only 3 `unsafe*` functions were added (not 23), yet all violations were resolved. Building what's needed, not what's speculative.
- **Peer layer architecture**: Foundry and Kit are now independent peers at Layer 1, both depending only on PLGG foundation. Application code composes them at the integration layer.
