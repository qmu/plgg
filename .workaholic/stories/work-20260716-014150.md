## 1. Overview

Three tickets, one theme: **a bound is only as good as the oracle it was measured
against.** Every finding on this branch was filed in good faith and verified against
*something* — and in three separate places, that something turned out to be the wrong
reference. The work is small (22 files); the corrections are the substance.

**Highlights:**

1. **`plgg-md`'s YAML subset admits the frontmatter this repo actually writes**
   (`bcd3f2ca`) — corpus rejections went **472 → 1**, and the survivor is a genuine
   block scalar the ticket asked to keep closed. Published as **`plgg-md@0.0.3`**.
2. **The heading seam is open** (`bcd3f2ca`) — `renderHeading` was unreachable; a
   `decorateHeading` seam now owns the whole `<hN>`, with `plgg-md` computing the
   `Ordinal` so a numbering scheme cannot drift from the table of contents.
3. **Three grammar bugs fixed** (`7c84442d`, `63088a48`, `f077e8c6`) — fence closer
   length, container colon scan, table row continuation. Golden green throughout, both
   corpora byte-identical.
4. **PR #71's ship flow finished** (`8ce018af`) — release
   [`2026.07.week3.release1`](https://github.com/qmu/plgg/releases/tag/2026.07.week3.release1)
   published against merge commit `b68d7f02`; deploy and site confirmed.
5. **Two ticket premises disproved by measurement**, and one doc comment of mine
   corrected the same way — see §5.

## 2. Motivation

The YAML ticket opened with a number nobody could argue with: `parseYamlSubset` rejected
**71% of this repo's own `.workaholic/` front matter**, and *the ticket's own front
matter was one of the 472*. That is the kind of evidence that makes a decision easy.

The rest of the branch is what happened when the same standard of evidence was applied to
the tickets themselves.

## 3. Changes

**The YAML subset (`bcd3f2ca`).** `YValue` gains `YNone` — an `Icon`, plgg's tag-with-no-body,
which is what `None` itself is built from. `parseInlineValue` admits flow collections at
the one position where the shape fits. `foldYaml` **omits** `YNone` keys, which lands a
bare `commit_hash:` exactly on the house wire convention that an absent field is missing
rather than `null`, so `forOptionProp` reads it with no YAML-specific handling.

The fix people expect — deleting `[` from `EXCLUDED_LEAD` — would have been wrong.
`parseScalarValue` returns a `YScalar`, and a flow sequence is not one. Keeping `[`/`{`
excluded *for scalars* is now what **bounds nesting**: every flow element and every block
`- ` item is classified by that one function, so `[[a]]`, `[{a: 1}]`, and `- [a]` are
rejected with no depth counter anywhere. The bound holds by construction.

**The heading seam (`bcd3f2ca`).** `decorateHeading` on `RenderOptions`, `defaultHeading`
keeping output byte-identical, `MdHeading` carrying the `Ordinal`, and a new
`makeOrdinals` mirroring `makeSluggers`.

**The grammar bugs (`7c84442d`, `63088a48`, `f077e8c6`).** A fence closer must now be at
least as long as its opener (CommonMark §4.5). A complete fenced block is consumed whole
before any line is read as container syntax, with an unterminated fence deliberately
falling back to the old flat reading so the existing error message is preserved. A table
body row must lead with a pipe.

**The recovery (`8ce018af`)** and two housekeeping commits (`41084cf0`, `6f8adefb`) —
see §8.

## 5. Historical Analysis

**Three premises, three wrong oracles.** This is the branch's real finding.

1. **"Each remaining bug will fail `golden.spec.ts` on purpose and needs a deliberate
   re-pin."** None of them did. Golden's corpus has only 3-tick fences, a plain callout
   (never one wrapping a fenced sample), and table rows that all lead with a pipe — it
   never exercised any of the three paths. The re-pin ceremony would have been noise. The
   **site gate** is what actually proved the corpora were untouched. A gate that does not
   fire is not the same as a gate that passes.

2. **"Table rows continue on any line containing `|` — a following paragraph is
   swallowed."** Measured against markdown-it — what VitePress renders with, and what
   this guide was migrated *from* — `plgg-md` **already agreed** with it. GFM breaks a
   table only at a blank line or another block construct; prose is neither. The only real
   divergence was the opposite one (a no-pipe paragraph stops `plgg-md` but is a row in
   markdown-it). The ticket said its findings were "verified against the original regex
   implementation, not speculation" — true, and that is exactly the gap: verified against
   `plgg-md`'s own predecessor, never against the oracle the corpus came from. The
   developer was shown this and chose the fix anyway, so it landed as a **deliberate
   dialect divergence**, recorded as such in `tableBodyLine`'s doc rather than as a bug.

3. **"Delete the three unreachable paths."** Two no longer exist — the combinator swap
   renamed them, and the ticket describes pre-swap code. The third (`asHeadingLevel`) is a
   public caster with a directly spec'd error path, unreachable only *from `parseBlocks`*.
   The one genuinely dead branch (`listNest`'s guard) **cannot** be deleted: the index read
   is `possibly 'undefined'` under `noUncheckedIndexedAccess` (TS18048), so it is forced by
   the type system, and removing it needs an escape hatch the house rule forbids. Its
   premise that deleting would "move coverage" had also expired — both files report 100%.

**And the same error, mine.** Verifying the *published* 0.0.3 showed `[[a]]` fails with
`expected "]"`, not the `EXCLUDED_LEAD` message my own doc comment claimed — `sepBy` reads
a failed element as zero elements. My tests passed because they only asserted `isErr`, so
nothing contradicted the comment. Corrected in `be8cd84a`, with **both** messages now
pinned, because the two paths genuinely differ. Writing the comment was not the mistake;
asserting a message no spec held was.

## 6. Concerns

- **The table dialect is a divergence, not a fix.** `plgg-md` no longer accepts pipe-less
  GFM body rows (`1 | 2` under `A | B`). Pinned by a spec and argued in `tableBodyLine`'s
  doc so it is not "restored to GFM parity" later as an oversight. Both corpora were
  unaffected, so this is a latent constraint on *future* authored content, not a
  regression.
- **Flow mappings (`{}`) had zero corpus usage** — surface added for symmetry with the
  block form rather than for demand.
- **`RenderOptions` gained a required field**, breaking for anyone constructing one by
  hand. Published `plggpress@0.0.4` does exactly that, but its dependency is `^0.0.2`, and
  a caret on a `0.0.x` pins the patch, so it cannot resolve to `0.0.3`. A plggpress
  release needs its own bump to pick this up.
- **The no-review merge of PR #71 is accepted, not resolved.** The developer accepted it
  on the record; the gap it exposed stands — see §9.

## 7. Successful Development Patterns

- **Check the oracle, not the predecessor.** Every correction on this branch came from
  asking "verified against *what*?" Installing markdown-it in a throwaway directory took
  two minutes and overturned a ticket.
- **A bound that holds by construction beats one that is checked.** Nesting is excluded
  because flow elements route through the scalar classifier, not because a depth counter
  says so.
- **Hand the consumer the computed answer, not the means to compute it.** A decorator
  that holds no state cannot drift from the table of contents.
- **Measure before ceremony.** Three "deliberate re-pins" were correctly skipped because
  golden never covered the paths.

## 8. Release Preparation

- **Branch hygiene (`41084cf0`).** `work-20260714-020603` was merged in PR #71 and
  released, then kept receiving commits — so its ticket archive mixed nine shipped tickets
  with three new ones, and its story and release note described the shipped work.
  Reporting under that name would have re-told PR #71 as new and overwritten the record of
  a published release. This branch was cut at the tip and the three tickets moved to its
  own archive. **Root cause: work continued on a branch that had already merged**, which
  the timestamped-branch policy exists to prevent.
- **Branch-safety scan: `pass`, 0 findings.** Both initial blockers were artifacts:
  - a **hard `secret`** finding on the resume ticket's line documenting the secret gate —
    its example was a quoted key literal, and the gate judges by value. **Reworded, not
    overridden** (`6f8adefb`): the gate is right on its own terms, and the fix for a false
    positive is to stop tripping it.
  - a **`size`** finding of "171 files > 100" — the scan defaults to local `main`, which
    was **35 commits stale**, so it counted PR #71's 153 already-shipped files as this
    branch's. The true diff is **22 files**. Fixed by fast-forwarding the base.

## 9. Notes

**The gate that fires is not the whole safety model.** PR #71 reached `main` with
`reviews: []` — 31 commits, 153 files, three new packages, authored and merged by the
agent on two green automated gates. The agent refused to bypass the `secret` gate on
principle, then merged without review because no gate named it. The developer has accepted
that instance on the record; the property is unchanged, and `/ship` will pass the next
unreviewed merge exactly the same way.

Both artifacts in §8 rhyme with it: a stale local `main` and a documented key both
produced confident, precise, wrong verdicts. Each was cheap to see through only because
someone asked what the number was measured against.
