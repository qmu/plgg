## 1. Overview

The branch delivers `plgg-token-metering`: token counting and cost estimation with **no dependency on any existing tokenizer library**, built from the published research article's calibration data (research repo, token-metering topic, merged 2026-07-17). The library is the second leg of the HQ `llm-token-metering` mission — the first leg (the article) supplies the method selection, the calibration parameters, the sample set, and the error table this package encodes.

The design center is honesty carried in types. `TokenEstimate` is a sum — `ExactTokenCount | EstimatedTokenCount` — so a caller must `match` rather than read a bare number that means different things per model. Every result carries `AccuracyEvidence` (the error band, holdout mean/max, whether the ±5% target was met, the samples version, and the source report), so **an accuracy claim cannot travel without its source**. The published truth is stated plainly and is test-enforced: OpenAI and Qwen count exactly (0.00%); **Claude (8.54% mean / 16.24% max) and Gemini (6.60% / 15.73%) are estimators and miss the ±5% target**. A test fails if that statement is ever softened.

**Highlights:**

1. `countTokens(input)(counter)` and `estimateCost(usage)(prices)`, both total, returning `Result` — plus `projectCost`, vocabulary parsers, and the Anthropic image-token formula
2. Branded types throughout: `TokenCount`, `UsdAmount`, `UsdPerMTok`, `ModelId`; a count becomes money only through `priceTokens`
3. `tokenBounds` inverts the band by **division** (`actual = predicted / (1 + e/100)`) — the intuitive `predicted × (1 ± band)` is wrong in both directions
4. `TokenCounter` construction enforces method selection: an exact counter for Claude is a typed `MethodMismatchError`
5. Vocabularies are **caller-supplied data**, not bundled I/O — the package passes the vendor-boundary gate conformant and unexempted, `dependencies` exactly `{ plgg }`
6. Every price entry requires `PriceProvenance`; missing rates return `MissingPriceError` rather than a substituted number

## 2. Motivation

The mission's premise is that per-principal usage metering needs counting that does not import a vendor's tokenizer. The research leg measured what is achievable: exact self-BPE where vocabularies are published, calibrated estimation where they are not, with the error bands published openly. This branch turns that measured result into a library whose API cannot misrepresent it — the estimator bands are not documentation, they are values the type system forces through every call.

Aggregation and storage are deliberately out of scope: the article's design chapter assigns them to the consumer, and this package stays a pure counting-and-pricing core.

## 3. Changes

`packages/plgg-token-metering/` (new), named from the article's own phrase to keep the cross-repo ubiquitous language. Layout `src/domain/{model,usecase}` with **no `vendors/`** — the package does no I/O. Wired into `build.sh`, `npm-install.sh`, `check-all.sh`, four `scripts/*-plgg-token-metering.sh`, the root README, and a new architecture audit row. Two four-point dependency judgments recorded in `DEPENDENCY-LOG.md`: tokenizer libraries **not taken** (none covers Anthropic's current models; native/WASM bindings; ~150-line loop over published data suffices; exit path = adopt a library behind the same `TokenCounter` interface per provider), and vocabulary data **caller-supplied, not bundled** (3.6 MB, perishable).

Prices: Claude $3/$15, GPT-5.5 $5/$30, Gemini $2/$12 from the research repo's `foundation-models` §7 (read 2026-07-17); Qwen $0.66 input from the run's own family card. **No cache rates for any model and no Qwen output rate are published anywhere the article measured — so none are shipped**; a non-zero bucket for them errors instead of substituting. The registry-price lesson from the same day's trend-recency repair (a 5×-stale $25→$5) is recorded, and the corrected $5 is asserted by a test.

Deliberate omissions where the article supplies no measurement: cache rates, Qwen output rate, and the **Google/OpenAI image conversions** — Google's documented 258 tokens/image contradicts the article's own probe of the same PNG (1,089, a 4.2× gap), so neither is shipped rather than inventing one. Anthropic's `w×h/750` **is** shipped: probe 124 = formula 120 + fitted overhead 4.

## 5. Historical Analysis

This is the mission's designed hand-off working as drawn: the article recorded calibration values, the sample set (`tm-v1`), and portable code precisely so a later session could build the library without re-measuring. The near-miss on the article side the same day — a fixture render showing ~1.3% error that would have implied ±5% met everywhere, caught before merge and regenerated from the measured frame — is the direct ancestor of this library's design rule that accuracy claims carry their evidence in the type.

## 6. Concerns

- The price table is perishable by nature; provenance fields say where and when each number came from, but nothing refreshes them automatically.
- The estimator calibrations are frozen to the article's sample set; when providers change tokenizers, the bands go stale silently until the research topic's recurring trial re-measures.
- One first attempt at a fixture guard silently no-op'd (Prettier had re-quoted the string); the unmasked test result exposed it. The committed guard was verified red-then-green.

## 7. Successful Development Patterns

- **Honesty as a test subject**: the README's accuracy statement is asserted by a test, so softening the miss into marketing fails CI.
- **Refusing to invent measurements**: three omissions (cache rates, Qwen output, image conversions) are recorded as absences that error, not defaults — the 258-vs-1089 contradiction is documented instead of resolved by fiat.
- **Reproduction against the real artifact**: the suite's fixtures are backed by a full re-run against the actual published vocabulary, not only recorded expectations.
