# Branch Story: work-20260623-214128

## 1. Overview

A from-the-design redesign of **plgg-test**, the family's own test
runner, so its public authoring/assertion API embodies plgg's core
idiom — pipe-style, data-last functional composition — instead of the
fluent `expect(x).toBe(y)` method chain of the first attempt. The test
suite is the most-read, most-written code in the repo; a fluent
throw-on-mismatch chain at the centre of an Option/Result, data-last,
exhaustively-matched codebase trains every reader that "the plgg way"
has an exception. This branch removes that exception: an assertion is
now a composable function you pipe a value through, returning a branded
plgg `Result`.

Delivered through a three-agent trip (Planner / Architect / Constructor,
Implosive Structure). Idiomatic composition was explicitly prioritised
**over** drop-in vitest/old-API migration: every spec was rewritten, not
re-imported, with the prior runner's verdicts as a precise oracle.

## 2. Highlights

- **Assertion = a branded `Result`** (`Result<Pass, Fail>`, tagged with
  plgg's own `Box`), so an assertion is a first-class citizen of
  `pipe`/`cast`/`proc`/`matchResult` — zero new vocabulary.
- **Data-last matchers**: `pipe(actual, toBe(expected))` /
  `check(actual, ...matchers)`; `all` aggregates *every* failure as
  siblings (the way `cast` accumulates validation errors); async is a
  `proc` step.
- **Narrowing is data-flow, not control-flow**: value-carrying matchers
  (`shouldBeOk`/`okThen`) unwrap the inner value into the pipeline
  (`cast(asInt("42"), shouldBeOk(), toBe(int(42)))`) — no `asserts cond`,
  no throw, and the old ambient `.d.ts` narrowing shim is deleted.
- **Anti-false-green guard**: a test body *returns* its assertion; the
  runner fails any body that returns void / a bare domain `Result` / a
  non-assertion (the brand is what makes a verdict distinguishable from
  data). A plain-`throw` meta-harness proves each drop-shape fails.
- Reused all style-agnostic plumbing (runner shell, discovery, reporter,
  `module.register`+`ts.transpileModule` execution, `fs.watch` watch,
  four-metric `NODE_V8_COVERAGE` coverage); only the assertion layer +
  the runner fold-seam changed. Runtime dependency stays **`plgg` only**.

## 3. Changes (journey)

This branch was built by an Agent Teams trip rather than discrete
tickets; the narrative is in the commits and the trip artifacts.

- **Planning** — concurrent direction / model / design, one-turn
  cross-review (all approvals). Consensus fixed nine build guardrails,
  the keystone being "the Assertion must be a branded Result or the
  anti-false-green guard is unbuildable."
- **Coding** — Constructor built the L0 assertion layer, the runner
  fold-seam, and the drop-shape meta-harness; rewrote all 74 plgg specs
  + the package self-tests to the pipe idiom; Architect reviewed,
  Planner ran E2E + parity + a mutation spot-check.
- **Consumer-side fixes (found after the package was clean):**
  1. plgg-test resolved to `any` from plgg — it is ESM (`type:module`),
     so under `module:NodeNext` plgg could not resolve plgg-test's
     extensionless dist `.d.ts` re-exports. Fixed by converting
     plgg-test's internal imports to relative NodeNext `./X.js` form
     (+ a loader-hook `.js`→`.ts` redirect so the CLI still runs from
     source).
  2. Matcher over-constraint — `toBe`/`toEqual` made the *actual*
     generic on application; `errThen` defaults `<E = InvalidError>`.
     No hand-annotation, no escape hatches.

## 4. Outcome

A complete, idiomatic, trustworthy test framework. The test API stops
*imitating* plgg and starts *being* plgg. plgg-test bumped to 0.0.2.

## 5. Historical analysis

This is the second plgg-test trip. The first (`work-20260623-170403`)
delivered a working runner with a vitest-compatible fluent API, optimised
for drop-in migration — which forced the throw-on-mismatch chain. That
API was rejected for being non-idiomatic; this branch reverses the
priority (idiom over migration) and supersedes it. The first trip's
style-agnostic plumbing was reused as prior art.

## 7. Successful development patterns

- **Branded-type keystone**: making the verdict a *branded* Result
  resolved the tension between "zero new vocabulary" and "no false
  green" — the single most important design decision, caught in review
  before a line was written.
- **Mutation spot-check beats bare parity**: verdict-parity (465/0)
  proves *same conclusions*; flipping expected values on a slice and
  requiring those tests to fail proves the rewrites have *force*.
- **Diagnosis-to-fix discipline**: the `any`-resolution and
  matcher-inference blockers were root-caused (ESM-dts/NodeNext;
  generic-callback inference) before fixing, so the fixes were
  API/packaging-level, not hand-annotation.
