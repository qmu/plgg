## 1. Overview

This branch makes two `plgg-auth` crypto tests runtime-portable so a red `main` goes green again. PR #54 (the four-phase OIDC provider) merged to `main`, but its fresh-clone `Run Tests` CI failed: two tests forced a signing failure by feeding a **structurally-valid but cryptographically degenerate RSA JWK** to the signer and asserting a rejection. That key imports lazily and only fails (or doesn't) at sign time, and *whether* it fails depends on the runtime's OpenSSL build — the GitHub Actions ubuntu runner signs it successfully, so both `expected Err` assertions fail there while passing locally.

The fix replaces the fragile bad-key-material trigger with a **deterministic, spec-mandated wrong-usage failure** that WebCrypto raises identically on every runtime, keeping both defensive branches covered. All gates are green on a fresh full rebuild.

**Highlights:**

1. **Portable crypto-failure trigger** — a `CryptoKey` that lacks the `"sign"` usage makes `crypto.subtle.sign` throw `InvalidAccessError`, and a JWK whose `use` contradicts the requested usage is rejected at import with `DataError`. Both are validation errors WebCrypto raises *before* any key-material math, so they are build-independent — unlike a degenerate key, whose acceptance is lazy and OpenSSL-build-specific.
2. **Genuine seam, not a test hook** — `signJws` is split into `importSignKey` → `signWith(cryptoKey)`, an honest import/sign decomposition that lets a spec drive a verify-only key into the sign path.
3. **One-field, RFC-legitimate model addition** — `RsaPrivateJwk` gains the optional RFC 7517 `use?: "sig" | "enc"`, emitted by `privateJwkJson` only when set, so ordinary signing keys are byte-for-byte unaffected.

## 2. Motivation

`main` was red on the `Run Tests` workflow — the fresh-clone `scripts/check-all.sh` backstop — which blocks the deferred PR #54 ship (GitHub Release + npm publish) and any further plgg-auth work. A plain local `npm run coverage` is green on this host, so the failure is invisible without reasoning about the runtime contract: the two tests relied on `crypto.subtle.sign` rejecting a garbage key, which is not a portable guarantee. Unblocking `main` required a trigger that fails deterministically on every runtime rather than one that happens to fail on the developer's machine.

## 5. Historical Analysis

The root cause was pre-recorded: the phase-1 and phase-3 `plgg-auth` Final Reports both flagged that "Node's WebCrypto validates RSA JWK material lazily — a garbage key can be accepted and only fails (or doesn't) at sign/verify, and the exact point differs across runtimes." This branch is where that known gotcha actually bit CI, and it converts the insight into a durable practice: the two tests that leaned on the garbage-key trick are exactly the two that failed on the runner. The `signWith` extraction follows the same import-then-operate decomposition the Jose layer already uses for verification (`importVerifyKey` → verify), and reuses the existing verify-only importer rather than adding new crypto surface.

## 7. Successful Development Patterns

- **Prefer a validation failure over a crypto failure for deterministic tests** — to force `crypto.subtle.sign` to reject on every runtime, use wrong key usage (`InvalidAccessError`) or a contradictory JWK `use`/`key_ops` (`DataError`), both checked before any key-material math. A structurally-valid-but-degenerate key is not portable: its acceptance is lazy and OpenSSL-build-specific.
- **Empirically pin the runtime contract before choosing a trigger** — a throwaway probe confirmed on this host that the garbage key imports OK and fails at sign with `OperationError` (the non-portable path), while a verify-only key throws `InvalidAccessError` (the portable path), turning a "should be portable" guess into a demonstrated fact.
- **Extract an honest seam instead of a test hook** — splitting `signJws` into `importSignKey` → `signWith` is a real decomposition (import and sign are genuinely two steps), so the test drives production code paths rather than a mock, and the reject arm lives in exactly one place.
- **Re-check the branch metric, not just line coverage** — plgg-auth branch coverage is sensitive to error-path structure; the conditional `use` emit and both defensive arms were verified against the branch number, not assumed from statements/lines.
