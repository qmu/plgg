# Design: Build the `plgg-ir` Package Family

> Full design document captured at mission creation (2026-07-11).
> This is the consolidated result of prior discussion; `mission.md` holds the
> distilled goal/scope/acceptance, this file holds the complete rationale.

## 1. Mission Overview

The mission is to design and implement a family of three packages that together provide a foundation for AI-generated, statically verifiable intermediate representations within the `plgg` ecosystem.

The package family will consist of:

```text
plgg-ir-syntax
plgg-ir-language
plgg-ir-manifest
```

These packages will allow an LLM-based AI agent to generate a restricted Lisp-style expression, validate its structure and meaning, normalize it into a canonical intermediate representation, and provide that representation to consumers such as `plggmatic`.

The initial use case is the generation of a standard Web application from a Domain Manifest. The generated system may include:

* data models
* database mappings
* validation rules
* authorization policies
* list and detail views
* navigation
* update actions
* aggregate boundaries
* derived values
* dependency ordering
* consistency rules

The broader goal is not limited to `plggmatic`. The same language-processing foundation should be reusable across the wider `plgg` family.

---

## 2. Background

The original problem began with the need to extract a domain model from an existing database structure.

The model must describe at least three categories of information:

1. **Persistence mapping**

   * Which domain concept corresponds to which table
   * Which domain field corresponds to which column

2. **Validation**

   * Which values are valid
   * Which constraints must be satisfied before registration or update

3. **Structure and update dependencies**

   * How entities relate to one another
   * In which order related records must be created or updated
   * How a change in one entity affects derived or materialized values in another entity

This representation was named the **Domain Manifest**.

A Domain Manifest is not simply a database schema, configuration file, or UI definition. It is a semantic description of the domain and its operational constraints.

It can serve as the common source for generating:

* validators
* APIs
* database adapters
* registration plans
* authorization checks
* UI components
* page structures
* documentation
* dependency graphs
* consistency enforcement logic

---

## 3. Why an Intermediate Representation

The Domain Manifest will primarily be generated by LLM-based AI agents rather than written directly by human developers.

The expression generated by the AI will then be consumed by deterministic libraries and applications such as `plggmatic`.

This makes the representation genuinely intermediate.

```text
Database Schema
Natural-Language Requirements
Existing Source Code
UI Requirements
          ↓
       AI Agent
          ↓
       plgg IR
          ↓
 plggmatic and other consumers
          ↓
Generated Web System
```

The term **IR** means **Intermediate Representation**.

An IR is a structured representation that exists between one or more input sources and one or more output systems. It is commonly used in compilers, code generators, static analyzers, and transformation pipelines.

In this project, the IR separates probabilistic AI generation from deterministic application behavior.

The AI produces a restricted representation. The consumer validates and interprets it.

The AI does not directly produce arbitrary React, TypeScript, SQL, or authorization code.

---

## 4. Why Lisp-Style S-Expressions

The IR will use Lisp-style S-expressions as its source representation.

For example:

```lisp
(entity project
  (field name
    (type string)
    (validate
      (required)
      (length-between 1 200))))
```

The purpose is not to implement a complete Lisp, Scheme, or Common Lisp runtime.

The representation borrows several useful structural properties from Lisp:

* prefix notation
* symbols
* lists
* uniform expression trees
* expressions that are directly represented as data
* easy structural transformation
* natural recursive processing
* compatibility with macro-like expansion
* a small and regular grammar

The value of Lisp-style syntax is especially clear when expressions contain conditions, derivations, references, or policies.

In YAML, a condition may have to be represented as a verbose object:

```yaml
requiredWhen:
  operator: equals
  left:
    reference: customerType
  right:
    literal: corporation
```

Alternatively, it may be embedded as an opaque string:

```yaml
requiredWhen: "customerType == 'corporation'"
```

In an S-expression, the condition is already represented as a syntax tree:

```lisp
(required-when
  (= customer-type corporation))
```

This expression can be parsed, type-checked, transformed, inspected, normalized, and translated without parsing a second language from a string.

---

## 5. What the Language Is Not

The language is not intended to be:

* a complete Lisp implementation
* a compatible subset of Scheme
* a general-purpose programming language
* an embedded scripting environment
* an arbitrary evaluator
* an application runtime
* a language with unrestricted side effects

The language should not expose operations such as:

```lisp
(eval ...)
(read-file ...)
(http-get ...)
(random)
(current-time)
```

Arbitrary I/O, mutation, nondeterminism, runtime evaluation, and unrestricted user-defined functions would undermine the main benefits of the IR:

* reproducibility
* static analysis
* security
* deterministic generation
* caching
* canonical serialization
* reliable diffing
* safe LLM output validation

The language may express behavior, but only symbolically and through a closed vocabulary.

---

## 6. The Core Design Principle

The IR should describe:

* what is valid
* what is related
* what is visible
* what is permitted
* what changes
* what depends on what
* what must remain consistent

It should avoid describing low-level execution steps wherever those steps can be derived.

For example, the IR should not generally say:

```lisp
(on-change order-item
  (update order.total ...))
```

Instead, it should express that `order.total` is derived from `order-items`.

```lisp
(field total
  (type (money JPY))
  (derive
    (sum
      (children order-items subtotal)))
  (materialize
    (consistency immediate)))
```

The consumer can then derive:

```text
order-item change
    ↓
subtotal recomputation
    ↓
order total recomputation
    ↓
materialized parent update
```

This keeps the IR declarative while still making operational behavior derivable.

## 7. The Domain Manifest

A Domain Manifest is a semantic artifact encoded in the IR.

A simplified example follows:

```lisp
(plgg-ir 1
  (module project-management

    (entity client
      (field id
        (type client-id))

      (field name
        (type string)
        (validate
          (required)
          (length-between 1 200)))

      (relation projects
        (target project)
        (cardinality many)
        (inverse client)))

    (entity project
      (field id
        (type project-id))

      (field name
        (type string)
        (validate
          (required)
          (length-between 1 200)))

      (relation client
        (target client)
        (cardinality one)
        (required)
        (inverse projects)))))
```

The Domain Manifest may include:

* entities
* fields
* primitive types
* domain-specific types
* relations
* persistence mappings
* validation
* invariants
* aggregates
* actions
* policies
* queries
* views
* navigation
* derived values
* materialized values
* consistency models

## 8. Type-Checked Domain Expressions

The IR should be statically typed.

This means operators, references, operands, and expression results must be verified before the representation is consumed.

For example:

```lisp
(field age
  (type integer))

(field name
  (type string))
```

This is valid:

```lisp
(>= age 18)
```

The operator has a type similar to:

```text
>= : Integer × Integer → Boolean
```

This is invalid:

```lisp
(>= name 18)
```

The type checker should report:

```text
Expected: Integer
Actual: String
```

The type system should preserve domain meaning rather than only database storage types.

For example:

```lisp
(field customer-id
  (type customer-id))

(field organization-id
  (type organization-id))
```

Even if both values are physically stored as strings, they should not be freely interchangeable.

The following expression may therefore be rejected:

```lisp
(= customer-id organization-id)
```

Parameterized domain types may also be supported:

```lisp
(field subtotal
  (type (money JPY)))

(field tax-rate
  (type percentage))

(field tax
  (type (money JPY))
  (derive
    (* subtotal tax-rate)))
```

A corresponding type rule may be:

```text
Money<C> × Percentage → Money<C>
```

The type system should also reject invalid operations such as:

```text
Money<JPY> + Money<USD>
Date + Money
String used as Boolean
Entity used as scalar
Collection used as single value
```

## 9. Validation Semantics

Validation answers the question:

> Is this value or entity state valid?

Validation may apply to:

* individual fields
* multiple fields
* entire entities
* action input
* post-update state

Field validation:

```lisp
(field email
  (type email-address)
  (validate
    (required)
    (max-length 254)))
```

Cross-field validation:

```lisp
(entity contract
  (field starts-at
    (type date))

  (field ends-at
    (type date))

  (invariant
    (before starts-at ends-at)))
```

Conditional validation:

```lisp
(field corporate-number
  (type corporate-number)
  (validate
    (required-when
      (= customer-type corporation))))
```

Validation should describe what is valid, not where validation is executed.

The same rule may later be interpreted as:

* browser validation
* API request validation
* application-service validation
* database constraint
* generated documentation
* test generation

---

## 10. Authorization Semantics

Authorization answers the question:

> Is this actor permitted to refer to or modify this data?

Read access and update access must be separate.

```lisp
(entity project
  (access
    (read project-reader)
    (update name project-name-editor)))
```

A project may be visible to a user while still being immutable to that user.

A policy may depend on:

* actor identity
* actor roles
* organization membership
* the current entity
* the previous state
* the proposed new state
* related entities

Example:

```lisp
(policy project-reader
  (allows
    (= actor.organization-id
       project.client.organization-id)))
```

Update policy:

```lisp
(policy project-name-editor
  (allows
    (and
      (= actor.organization-id
         project.client.organization-id)

      (has-role actor project-manager))))
```

Actions should explicitly reference policies:

```lisp
(action edit-project-name
  (subject project)

  (input
    (field name
      (type string)
      (validate
        (required)
        (length-between 1 200))))

  (authorize
    (policy project-name-editor))

  (effect
    (set project.name input.name)))
```

Authorization must be enforced by every execution path.

It must not exist only as a UI condition.

A UI consumer may hide or disable the edit action, but an API or application service must still reject unauthorized execution.

The default behavior should be:

```text
No authorization policy = denied
```

This is safer than assuming permission when authorization is omitted.

## 11. Views and Navigation

The IR must be capable of describing a standard Web system.

For example:

1. A client detail page shows projects.
2. Selecting a project opens the project detail page.
3. An unauthorized user cannot update the project name.

Client detail view:

```lisp
(view client-detail
  (subject
    (entity client)
    (parameter client-id))

  (query
    (one client
      (where
        (= client.id client-id)))

    (include
      client.projects))

  (layout
    (detail
      (show client.name)

      (section projects
        (list client.projects
          (show project.name)

          (navigate
            (to project-detail)
            (with
              (project-id project.id))))))))
```

Project detail view:

```lisp
(view project-detail
  (subject
    (entity project)
    (parameter project-id))

  (query
    (one project
      (where
        (= project.id project-id))
      (authorized-by project-reader)))

  (layout
    (detail
      (show project.name)
      (show project.client.name)
      (action edit-project-name))))
```

Navigation should refer to semantic views and typed parameters rather than hard-coded URLs.

```lisp
(navigate
  (to project-detail)
  (with
    (project-id project.id)))
```

A Web consumer may translate this into a URL route. Another consumer may translate it into a mobile navigation action.

## 12. Actions and Effects

An action represents a domain operation available to a user or system.

It may contain:

* a subject
* typed input
* validation
* authorization
* effects
* postconditions
* consistency requirements

Example:

```lisp
(action edit-project-name
  (subject project)

  (input
    (field name
      (type string)
      (validate
        (required)
        (length-between 1 200))))

  (authorize
    (policy project-name-editor))

  (effect
    (set project.name input.name))

  (ensure
    (valid project)))
```

The expected processing order is:

```text
1. Resolve subject
2. Resolve actor
3. Check authorization
4. Validate input
5. Apply effect
6. Check invariants and postconditions
7. Persist changes
8. Update derived or materialized values
```

The IR should represent the semantic operation, while consumers determine concrete implementation details.

---

## 13. Aggregates and Update Dependencies

The language must express how related entities form consistency boundaries.

Example:

```lisp
(aggregate project-aggregate
  (root project)
  (members task)
  (consistency immediate))
```

This declares that:

* `project` is the aggregate root
* `task` belongs to the aggregate
* updates within the aggregate require immediate consistency

Derived parent values should be expressed through dependencies.

For example:

```lisp
(entity project
  (field task-count
    (type integer)

    (derive
      (count project.tasks))

    (materialize
      (consistency immediate))))
```

When a task is added or removed, the dependency graph implies:

```text
project.tasks changed
    ↓
project.task-count invalidated
    ↓
project.task-count recomputed
    ↓
project record updated
```

The update order should be derived from the dependency graph.

Numeric update ordering should generally be avoided.

Bad:

```lisp
(order 10)
(order 20)
```

Preferred:

```lisp
(derive
  (count project.tasks))
```

The system can perform topological sorting and detect dependency cycles.

## 14. Structural Static Verification

The IR must detect structural inconsistencies before execution.

Consider the following entity structure:

```text
Client
  ├── Projects
  │     └── Tasks
  └── Invoices
```

Example definitions:

```lisp
(entity client
  (relation projects
    (target project)
    (cardinality many))

  (relation invoices
    (target invoice)
    (cardinality many)))

(entity project
  (relation client
    (target client)
    (cardinality one))

  (relation tasks
    (target task)
    (cardinality many)))

(entity task
  (relation project
    (target project)
    (cardinality one)))

(entity invoice
  (relation client
    (target client)
    (cardinality one)))
```

The following reference is immediately invalid:

```lisp
(show task.invoices)
```

There is no direct `task.invoices` relation.

The compiler should report:

```text
Unknown relation: task.invoices
```

However, this path is structurally reachable:

```lisp
task.project.client.invoices
```

Therefore, relation existence alone is not enough to determine whether a view may use the data.

The language must distinguish:

1. **Entity graph**

   * What is structurally related

2. **Aggregate boundary**

   * What belongs to the same consistency boundary

3. **Query scope**

   * What the view explicitly loaded

4. **Projection boundary**

   * Which external fields were intentionally exposed

5. **Authorization scope**

   * Which data the actor may access

Example:

```lisp
(view task-detail
  (subject task)
  (scope project-aggregate)

  (query
    (one task)
    (include task.project))

  (layout
    (show task.name)
    (show task.project.name)))
```

The following reference should be rejected:

```lisp
(show task.project.client.invoices)
```

Possible diagnostics:

```text
The path is not included in the view query.

The path crosses the project aggregate boundary.

The invoices collection cannot be traversed through an external reference.

The target is not exposed through an approved projection.
```

This allows the language to enforce architectural boundaries rather than merely checking whether a SQL join is possible.

---

## 15. Query Scope and Projections

A view should only be able to use values present in its declared query scope.

For example:

```lisp
(query
  (one task)
  (include task.project))
```

This exposes:

```text
task
task.project
```

It does not expose:

```text
task.project.client
task.project.client.invoices
```

If external data is required, it should be introduced explicitly through a projection.

```lisp
(projection client-summary
  (from client)
  (fields
    client.id
    client.name))
```

A view may then request:

```lisp
(lookup client-summary
  (through task.project.client))
```

The result does not contain invoices.

Therefore:

```lisp
(show client-summary.invoices)
```

must fail static verification.

This makes cross-boundary access deliberate and reviewable.

## 16. Required Static Verification Passes

The `plgg-ir` toolchain should perform at least the following checks.

### 16.1 Syntax validation

* malformed lists
* invalid literals
* incomplete expressions
* invalid tokens

### 16.2 Keyword and form validation

* unknown forms
* invalid child forms
* duplicated exclusive forms
* missing required forms

### 16.3 Name resolution

* unknown entities
* unknown fields
* unknown relations
* unknown views
* unknown actions
* unknown policies
* unknown projections

### 16.4 Type checking

* invalid operators
* invalid comparisons
* invalid assignments
* invalid action effects
* wrong query result types
* invalid Boolean conditions
* incorrect parameter passing

### 16.5 Relation verification

* invalid inverse relations
* incorrect cardinality pairs
* unresolved relation targets
* impossible ownership structures

### 16.6 Aggregate verification

* missing aggregate roots
* duplicate aggregate membership
* invalid root-member relations
* illegal aggregate traversal
* conflicting consistency declarations

### 16.7 Query and view verification

* use of unloaded relation paths
* missing route parameters
* unavailable fields
* invalid list and scalar usage
* unauthorized query paths

### 16.8 Authorization verification

* update actions without policies
* policies returning non-Boolean values
* invalid actor references
* unauthorized field updates
* policy references to unavailable data

### 16.9 Dependency verification

* circular derivations
* impossible materialization order
* conflicting sources of truth
* incompatible consistency requirements
* updates crossing transaction boundaries incorrectly

### 16.10 Canonicalization

* shorthand expansion
* default removal or explicit completion
* stable ordering
* resolved identifiers
* canonical type representation
* normalized references

## 17. Package Architecture

The implementation should be split into three packages.

```text
plgg-ir-syntax
plgg-ir-language
plgg-ir-manifest
```

The `plgg-ir` prefix defines a coherent package family for AI-generated intermediate representations.

---

# Package 1: `plgg-ir-syntax`

## 18. Purpose

`plgg-ir-syntax` provides the lowest-level syntax representation and parser infrastructure.

It knows how to read and write the source structure, but it does not understand domain-specific meaning.

It does not know what `entity`, `relation`, `aggregate`, or `policy` mean.

## 19. Responsibilities

The package should contain:

* tokenizer
* parser
* serializer or printer
* symbols
* strings
* numbers
* booleans
* lists
* syntax tree
* source positions
* source ranges
* syntax diagnostics

Example input:

```lisp
(entity project
  (relation client))
```

Example syntax tree:

```text
List
├── Symbol("entity")
├── Symbol("project")
└── List
    ├── Symbol("relation")
    └── Symbol("client")
```

## 20. Non-responsibilities

`plgg-ir-syntax` should not contain:

* entity semantics
* type checking
* name resolution
* aggregate validation
* authorization logic
* Domain Manifest models
* application generation

## 21. Package Description

> `plgg-ir-syntax` is a minimal S-expression syntax library for the `plgg-ir` family. It parses source text into position-aware syntax trees and serializes syntax trees back into a canonical textual representation without assigning domain-specific meaning.

---

# Package 2: `plgg-ir-language`

## 22. Purpose

`plgg-ir-language` provides reusable infrastructure for constructing restricted, statically analyzed IR languages on top of `plgg-ir-syntax`.

This package provides semantic machinery but does not define Domain Manifest vocabulary.

## 23. Responsibilities

The package should contain:

* form definitions
* operator definitions
* grammar constraints above raw S-expression syntax
* lexical and semantic scopes
* name resolution
* reference resolution
* type representations
* type inference where appropriate
* type checking
* semantic diagnostics
* validation passes
* normalization passes
* canonicalization
* dialect composition
* extension registration
* pass orchestration

Conceptual API:

```ts
type FormDefinition<Node> = Readonly<{
  readonly name: string
  readonly analyze: (
    syntax: Syntax,
    context: AnalysisContext,
  ) => Result<Node, ReadonlyArray<Diagnostic>>
}>
```

Typed operators may be registered conceptually as:

```ts
defineOperator({
  name: "and",
  parameters: [booleanType, booleanType],
  result: booleanType,
})
```

## 24. Dialect Composition

The package should support composing restricted dialects.

```ts
const language = compose(
  coreDialect,
  domainDialect,
  viewDialect,
  policyDialect,
)
```

This does not mean dynamically evaluating arbitrary extensions.

A dialect is a statically registered set of:

* forms
* types
* operators
* scopes
* analysis rules
* normalization rules

## 25. Non-responsibilities

`plgg-ir-language` should not define:

* client
* project
* task
* entity
* field
* relation
* aggregate
* view
* policy
* action

It should only provide the machinery required to define and analyze them.

## 26. Package Description

> `plgg-ir-language` is a reusable static language-processing framework built on `plgg-ir-syntax`. It provides form registration, name resolution, type checking, semantic diagnostics, normalization, canonicalization, and dialect composition for restricted intermediate-representation languages.

---

## 27. Purpose

`plgg-ir-manifest` defines the Domain Manifest vocabulary, semantics, static verification rules, and canonical intermediate model.

It is the domain-specific layer.

It uses `plgg-ir-language` to interpret syntax trees produced by `plgg-ir-syntax`.

## 31. Package Relationships

The dependency structure must remain one-directional.

```text
plgg
  ↑
plgg-ir-syntax
  ↑
plgg-ir-language
  ↑
plgg-ir-manifest
  ↑
plggmatic
```

More precisely:

```text
plgg-ir-syntax
  provides source structure

plgg-ir-language
  provides reusable semantic machinery

plgg-ir-manifest
  provides Domain Manifest semantics

plggmatic
  consumes the verified canonical manifest
```

The package dependency relationship is:

```text
plgg-ir-language depends on plgg-ir-syntax

plgg-ir-manifest depends on:
  plgg-ir-language
  plgg-ir-syntax

plggmatic depends on:
  plgg-ir-manifest
```

The lower-level packages must never import the upper-level packages.

## 32. Processing Pipeline

The intended pipeline is:

```text
LLM-generated source
        ↓
plgg-ir-syntax
        ↓
Position-aware syntax tree
        ↓
plgg-ir-language
        ↓
Generic resolution, type checking,
diagnostics, and normalization
        ↓
plgg-ir-manifest
        ↓
Domain-specific verification and
canonical Domain Manifest IR
        ↓
plggmatic
        ↓
Generated Web system
```

A more precise description is:

```text
Source Text
  ↓ parse

Syntax Tree
  ↓ expand
  ↓ resolve
  ↓ type-check
  ↓ verify
  ↓ normalize

Canonical Domain Manifest IR
  ↓ interpret

UI / API / Validation / Authorization /
Persistence / Navigation / Documentation
```

## 33. Canonical Representation

The source format may support limited shorthand, but the canonical IR must be explicit.

Source shorthand:

```lisp
(email-field email)
```

Expanded form:

```lisp
(field email
  (type email-address)
  (validate
    (required)
    (email-address)))
```

Canonical IR should contain:

* resolved entity IDs
* resolved field IDs
* resolved relation IDs
* explicit types
* explicit cardinality
* explicit policies
* explicit dependency references
* normalized expressions
* stable ordering
* version information

The canonical IR should be deterministic.

Equivalent source expressions should normalize to the same canonical representation.

This enables:

* stable caching
* semantic diffing
* content hashing
* reproducible generation
* artifact comparison
* easier testing
* safer LLM correction loops

---

## 34. LLM-Oriented Design

Because LLM agents will be the primary producers, the language should be optimized for reliable generation rather than manual convenience.

The design should favor:

* a small closed vocabulary
* regular expression shapes
* explicit references
* explicit versions
* minimal context-dependent syntax
* few implicit defaults
* one preferred representation per meaning
* precise diagnostics
* easy partial correction
* canonical formatting

The language should reject unknown forms.

```text
Unknown keyword = compilation error
```

It should not silently ignore unsupported fields or forms.

A versioned source may begin with:

```lisp
(plgg-ir 1
  (module project-management
    ...))
```

Versioning is required because AI prompts, consumers, and canonical models must agree on semantics.

---

## 35. Error Model

Diagnostics should include:

* error code
* severity
* message
* source range
* expected form or type
* actual form or type
* related source locations
* suggested correction where deterministic

Example:

```text
Code:
  manifest.view.relation-not-loaded

Message:
  The relation path task.project.client is not available
  in the query scope of view task-detail.

Location:
  line 48, column 11

Available paths:
  task
  task.project
```

For LLM correction loops, structured diagnostics should also be available.

```ts
type Diagnostic = Readonly<{
  readonly code: string
  readonly severity: "error" | "warning"
  readonly message: string
  readonly range: SourceRange
  readonly related: ReadonlyArray<RelatedDiagnostic>
}>
```

The exact TypeScript design should follow existing `plgg` conventions and avoid unnecessary mutable or unsafe constructs.

## 36. Safety Principles

The implementation should follow these principles.

### 36.1 Deny by default

Missing authorization does not imply permission.

### 36.2 No arbitrary evaluation

All executable-looking expressions are symbolic and interpreted through registered semantics.

### 36.3 Closed vocabulary

Unknown forms and operators are rejected.

### 36.4 Typed references

Entity, field, relation, action, policy, view, and projection references must not be interchangeable.

### 36.5 Explicit boundary crossing

Aggregate, query, and projection boundaries must be crossed intentionally.

### 36.6 Derived values declare sources

Materialized fields must identify their derivation rather than being independently writable.

### 36.7 Canonical source of truth

A value must not have conflicting authoritative sources.

### 36.8 Dependency cycles are errors

Cycles are rejected unless a future feature explicitly provides a valid fixed-point semantic model.

### 36.9 Consumer independence

Domain semantics must not directly encode React components, SQL fragments, URLs, or framework-specific implementation details.

---

## 37. Consumer Independence

The IR should describe meaning rather than implementation.

Avoid:

```lisp
(react-component "MuiTextField")
```

Prefer:

```lisp
(input
  (role email)
  (value-type email-address)
  (interaction editable))
```

A Web consumer may map this to one component. A mobile consumer may map it to another.

Similarly, avoid hard-coded routes:

```lisp
(url "/projects/:id")
```

Prefer semantic navigation:

```lisp
(navigate
  (to project-detail)
  (with
    (project-id project.id)))
```

The IR remains stable even when the rendering or execution technology changes.

* clients
* projects
* tasks
* invoices
* project-name editing
* authorization
* client detail view
* project detail view
* task detail view

```lisp
(view client-detail

    ...

      (to project-detail)

        (project-id project.id)))))
```

It should allow:

```text
A user can read a project.
A user can navigate from a client to a project.
A permitted user can edit a project name.
```

It should reject an unauthorized update:

```text
A user without project-name-editor permission

```

```lisp
(view task-detail
  ...
  (show task.project.client.invoices))
```

when invoices are outside the declared query, projection, or aggregate scope.

## 42. Naming Summary

The final package names are:

```text
plgg-ir-syntax
plgg-ir-language
plgg-ir-manifest
```

Their relationship can be summarized as:

```text
plgg-ir-syntax provides structure.

plgg-ir-language provides reusable
semantic-analysis machinery.

plgg-ir-manifest provides Domain Manifest
vocabulary and domain-specific semantics.
```

A longer description is:

> The `plgg-ir` family provides a shared foundation for AI-generated intermediate representations. `plgg-ir-syntax` parses and represents Lisp-style S-expression source, `plgg-ir-language` provides reusable static language-processing facilities such as name resolution, type checking, diagnostics, and normalization, and `plgg-ir-manifest` defines the Domain Manifest semantics used to describe entities, relations, validation, authorization, aggregates, views, actions, derivations, and consistency rules.

---

## 43. Final Definition

The overall system can be defined as follows:

> The `plgg-ir` package family is a restricted, typed, Lisp-style intermediate-representation platform for the `plgg` ecosystem. It enables LLM-based agents to generate statically verifiable domain expressions that can be normalized into a canonical representation and consumed deterministically by systems such as `plggmatic`. The initial Domain Manifest dialect expresses persistence mappings, domain types, validation, authorization, relations, aggregate boundaries, queries, views, actions, derivations, materialization, and update dependencies while preventing arbitrary execution and structural inconsistency.

The purpose of the mission is not merely to create another configuration language.

The mission is to create a safe semantic boundary between AI-generated intent and deterministic Web-system generation.
